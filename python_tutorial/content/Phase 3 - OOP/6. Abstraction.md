# 6. Abstraction

## Definition

**Abstraction** is the concept of hiding complex implementation details and exposing only essential features. In Python, abstraction is achieved through **Abstract Base Classes (ABCs)** using the `abc` module. An abstract class defines a blueprint (interface) that subclasses must implement — it cannot be instantiated directly.

```python
from abc import ABC, abstractmethod

class Shape(ABC):           # Abstract class
    @abstractmethod
    def area(self):         # Abstract method — must be overridden
        pass

# s = Shape()              # TypeError: Can't instantiate abstract class

class Circle(Shape):
    def __init__(self, radius):
        self.radius = radius

    def area(self):         # Must implement area()
        return 3.14 * self.radius ** 2
```

## Why it exists

Abstraction exists to:
- **Define contracts** — enforce that subclasses implement specific methods
- **Reduce complexity** — users interact with high-level interfaces, not internal details
- **Improve maintainability** — changes to implementation don't affect code using the interface
- **Enable polymorphism** — treat different implementations through a common abstract interface
- **Guide design** — provide a clear template for what subclasses should do

## How it works

Create an abstract class by inheriting from `ABC` and decorating methods with `@abstractmethod`. Any subclass must implement all abstract methods, or it too becomes abstract and cannot be instantiated.

```python
from abc import ABC, abstractmethod

class Database(ABC):
    @abstractmethod
    def connect(self):
        pass

    @abstractmethod
    def query(self, sql):
        pass

    @abstractmethod
    def disconnect(self):
        pass

class MySQLDatabase(Database):
    def connect(self):
        return "Connected to MySQL"

    def query(self, sql):
        return f"MySQL executing: {sql}"

    def disconnect(self):
        return "Disconnected from MySQL"

# class Incomplete(MySQLDatabase):
#     pass
# db = Incomplete()  # TypeError — abstract methods not implemented
```

## Internal working

### ABC and ABCMeta

`ABC` is a helper class that uses `ABCMeta` as its metaclass. The metaclass tracks which methods are abstract and prevents instantiation of classes with unimplemented abstract methods.

```python
from abc import ABCMeta

class Shape(metaclass=ABCMeta):   # Same as inheriting ABC
    @abstractmethod
    def area(self):
        pass
```

When you try to instantiate an abstract class, Python's metaclass checks if all abstract methods have been implemented:

```python
try:
    s = Shape()  # Raises TypeError
except TypeError as e:
    print(e)  # Can't instantiate abstract class Shape with abstract method area
```

### Abstract properties

You can also make properties abstract:

```python
from abc import ABC, abstractmethod

class Employee(ABC):
    @property
    @abstractmethod
    def salary(self):
        pass

    @salary.setter
    @abstractmethod
    def salary(self, value):
        pass

class Developer(Employee):
    @property
    def salary(self):
        return self._salary

    @salary.setter
    def salary(self, value):
        if value < 0:
            raise ValueError("Salary cannot be negative")
        self._salary = value
```

## Real-world analogy

Think of a USB port. It defines a standard interface (the USB specification) that any device must follow. A USB port doesn't care whether you plug in a mouse, keyboard, flash drive, or phone — any device that follows the USB standard works. The USB specification is the **abstract class**, and each device is a **concrete implementation**. You, the user, interact through the USB interface without knowing (or caring) about the device's internal electronics.

## Examples (with code)

### Example 1: Basic abstract class

```python
from abc import ABC, abstractmethod

class Animal(ABC):
    @abstractmethod
    def sound(self):
        pass

    @abstractmethod
    def move(self):
        pass

    def sleep(self):            # Concrete method — inherited as-is
        return "Zzz..."

class Dog(Animal):
    def sound(self):
        return "Woof!"

    def move(self):
        return "Runs on four legs"

class Fish(Animal):
    def sound(self):
        return "..."

    def move(self):
        return "Swims"

d = Dog()
f = Fish()
print(d.sound())    # Woof!
print(d.sleep())    # Zzz...
print(f.move())     # Swims

# a = Animal()      # TypeError!
```

### Example 2: Payment processor system

```python
from abc import ABC, abstractmethod

class PaymentProcessor(ABC):
    @abstractmethod
    def process_payment(self, amount):
        pass

    @abstractmethod
    def refund(self, transaction_id):
        pass

    def receipt(self, amount):
        return f"Payment of ${amount:.2f} processed."

class CreditCardProcessor(PaymentProcessor):
    def process_payment(self, amount):
        return f"Charged ${amount:.2f} to credit card."

    def refund(self, transaction_id):
        return f"Refunded transaction #{transaction_id} to credit card."

class PayPalProcessor(PaymentProcessor):
    def process_payment(self, amount):
        return f"Processed ${amount:.2f} via PayPal."

    def refund(self, transaction_id):
        return f"Refunded transaction #{transaction_id} via PayPal."

    def receipt(self, amount):
        return f"PayPal receipt: ${amount:.2f} sent to email."

def checkout(processor, amount):
    print(processor.process_payment(amount))
    print(processor.receipt(amount))

cc = CreditCardProcessor()
pp = PayPalProcessor()

checkout(cc, 50.00)
# Charged $50.00 to credit card.
# Payment of $50.00 processed.

checkout(pp, 75.00)
# Processed $75.00 via PayPal.
# PayPal receipt: $75.00 sent to email.
```

### Example 3: Abstract properties

```python
from abc import ABC, abstractmethod

class Appliance(ABC):
    @property
    @abstractmethod
    def power_consumption(self):
        """Watts per hour."""
        pass

    @abstractmethod
    def turn_on(self):
        pass

    @abstractmethod
    def turn_off(self):
        pass

class WashingMachine(Appliance):
    @property
    def power_consumption(self):
        return 500  # watts

    def turn_on(self):
        return "Washing machine started"

    def turn_off(self):
        return "Washing machine stopped"

class Refrigerator(Appliance):
    @property
    def power_consumption(self):
        return 150  # watts

    def turn_on(self):
        return "Refrigerator cooling"

    def turn_off(self):
        return "Refrigerator stopped"

appliances = [WashingMachine(), Refrigerator()]
for a in appliances:
    print(f"Power: {a.power_consumption}W")
    print(a.turn_on())
```

### Example 4: Multiple abstract methods with type hints

```python
from abc import ABC, abstractmethod

class Vehicle(ABC):
    def __init__(self, brand, model):
        self.brand = brand
        self.model = model

    @abstractmethod
    def start(self) -> str:
        pass

    @abstractmethod
    def drive(self, distance: float) -> str:
        pass

    @abstractmethod
    def stop(self) -> str:
        pass

    def info(self) -> str:
        return f"{self.brand} {self.model}"

class Car(Vehicle):
    def start(self) -> str:
        return "Engine started"

    def drive(self, distance: float) -> str:
        return f"Drove {distance} km"

    def stop(self) -> str:
        return "Engine stopped"

class ElectricScooter(Vehicle):
    def start(self) -> str:
        return "Motor activated"

    def drive(self, distance: float) -> str:
        return f"Glided {distance} km silently"

    def stop(self) -> str:
        return "Regenerative braking applied"
```

### Example 5: Abstract class with concrete helper methods

```python
from abc import ABC, abstractmethod
from datetime import datetime

class Logger(ABC):
    @abstractmethod
    def write(self, message: str):
        pass

    def log(self, level: str, message: str):
        """Concrete helper — uses abstract write()."""
        timestamp = datetime.now().isoformat()
        formatted = f"[{timestamp}] [{level}] {message}"
        self.write(formatted)

    def info(self, message: str):
        self.log("INFO", message)

    def error(self, message: str):
        self.log("ERROR", message)

class FileLogger(Logger):
    def __init__(self, filename):
        self.filename = filename

    def write(self, message: str):
        with open(self.filename, "a") as f:
            f.write(message + "\n")

class ConsoleLogger(Logger):
    def write(self, message: str):
        print(f"CONSOLE: {message}")

# Usage
logger = ConsoleLogger()
logger.info("Application started")
logger.error("Something went wrong")
# CONSOLE: [2026-01-01T12:00:00] [INFO] Application started
# CONSOLE: [2026-01-01T12:00:00] [ERROR] Something went wrong
```

## Common mistakes

1. **Forgetting to call `abstractmethod` on all abstract methods** — if even one abstract method is unimplemented, the subclass can't be instantiated
2. **Trying to instantiate an abstract class** — raises `TypeError`
3. **Forgetting `@abstractmethod` decorator** — without it, the method is just a regular method, and subclasses aren't forced to implement it
4. **Confusing ABCs with interfaces** — ABCs can have concrete methods + state; pure interfaces only have method signatures
5. **Overusing abstraction** — don't create abstract classes for everything; use them when you genuinely need a contract
6. **Name collision between abstract and concrete method** — Python doesn't prevent you from having the same method name in abstract and concrete forms but it defeats the purpose

## Interview questions + answers

**Q1: What is an abstract class?**
A: A class that cannot be instantiated and is designed to be subclassed. It contains abstract methods (declared but not implemented) that subclasses must implement.

**Q2: How do you create an abstract class in Python?**
A: Import `ABC` and `abstractmethod` from the `abc` module, inherit from `ABC`, and decorate abstract methods with `@abstractmethod`.

**Q3: Can an abstract class have concrete methods?**
A: Yes — abstract classes can have fully implemented methods that subclasses inherit. This lets you provide shared functionality alongside the contract.

**Q4: What happens if a subclass doesn't implement all abstract methods?**
A: The subclass remains abstract and cannot be instantiated. It raises `TypeError: Can't instantiate abstract class...`

**Q5: What is the difference between abstraction and encapsulation?**
A: Abstraction hides *complexity* and shows only essential features (what it does). Encapsulation hides *data* and bundles it with methods (how it does it). Abstraction is about design; encapsulation is about protection.

## Revision notes (bullet points summary)

- Abstract classes cannot be instantiated — they define contracts
- Import `ABC` and `abstractmethod` from `abc`
- `class MyClass(ABC):` to create an abstract class
- `@abstractmethod` decorator marks methods as abstract
- All abstract methods must be implemented in concrete subclasses
- Abstract classes can have concrete methods and `__init__`
- Abstract properties: `@property` above `@abstractmethod`
- ABCs ensure consistency across implementations
- Use abstraction to define "what must be done", not "how to do it"
- Python's ABC is the equivalent of interfaces in other languages
