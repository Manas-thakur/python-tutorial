# 3. Encapsulation

## Definition

**Encapsulation** is the bundling of data (attributes) and methods that operate on that data within a single unit (class), while restricting direct access to some of the object's internal components. In Python, encapsulation is achieved through naming conventions and property decorators — not through strict access modifiers like in Java or C++.

```python
class BankAccount:
    def __init__(self, owner, balance):
        self.owner = owner       # Public
        self._branch = "Main"    # Protected (convention)
        self.__balance = balance # Private (name mangling)

    def get_balance(self):
        return self.__balance
```

## Why it exists

Encapsulation exists to:
- **Protect data integrity** — prevent external code from putting objects into invalid states
- **Hide implementation details** — users interact through a clean interface, not internal complexity
- **Allow controlled modification** — validate changes before applying them
- **Make code maintainable** — internal changes don't break external code using the public interface

## How it works

Python has three levels of attribute visibility, all based on **naming conventions** (not enforced by the language):

| Convention | Name | Example | Accessibility |
|------------|------|---------|---------------|
| Public | `name` | `self.name` | Accessible from anywhere |
| Protected | `_name` | `self._age` | Convention: internal use only (still accessible) |
| Private | `__name` | `self.__salary` | Name mangled to `_ClassName__name` |

```python
class Employee:
    def __init__(self, name, salary):
        self.name = name          # Public
        self._department = "IT"   # Protected
        self.__salary = salary    # Private

e = Employee("Alice", 50000)
print(e.name)           # Alice — OK
print(e._department)    # IT — works but frowned upon
# print(e.__salary)     # AttributeError!
print(e._Employee__salary)  # 50000 — name mangled access
```

## Internal working

### Name mangling

When Python encounters an attribute starting with `__` (but not ending with `__`), it renames it to `_ClassName__attribute`:

```python
class Demo:
    def __init__(self):
        self.__secret = 42

    def reveal(self):
        return self.__secret  # Inside class, name is unmangled

d = Demo()
print(d.__dict__)              # {'_Demo__secret': 42}
print(d._Demo__secret)         # 42 — technically accessible
# print(d.__secret)           # AttributeError
```

Name mangling happens at class definition time. It's not a security mechanism — it's a **name collision prevention** mechanism for inheritance.

### Attribute access order with properties

```python
class Temperature:
    def __init__(self, celsius):
        self._celsius = celsius

    @property
    def celsius(self):
        """Getter — called when accessing temp.celsius"""
        return self._celsius

    @celsius.setter
    def celsius(self, value):
        """Setter — called when assigning temp.celsius = value"""
        if value < -273.15:
            raise ValueError("Temperature below absolute zero!")
        self._celsius = value

t = Temperature(25)
print(t.celsius)    # 25 — calls getter
t.celsius = 30      # calls setter
# t.celsius = -300  # ValueError!
```

## Real-world analogy

Think of encapsulation like a vending machine. You interact through a public interface (buttons, coin slot, dispenser tray). You don't directly access the internal inventory, cooling system, or payment processor. The machine protects its internal state and only lets you interact through approved channels. Even though you could theoretically pry open the machine and grab a snack, you're not supposed to — and doing so breaks the system's integrity.

## Examples (with code)

### Example 1: Public, protected, and private attributes

```python
class Person:
    def __init__(self, name, age, ssn):
        self.name = name           # Public — anyone can see/modify
        self._age = age            # Protected — "internal use"
        self.__ssn = ssn           # Private — name mangled

    def get_ssn(self):
        """Controlled access to private data."""
        return f"***-**-{self.__ssn[-4:]}"

    def set_age(self, age):
        if 0 < age < 150:
            self._age = age
        else:
            raise ValueError("Invalid age")

p = Person("Bob", 30, "123-45-6789")
print(p.name)          # Bob
print(p._age)          # 30 (accessible but discouraged)
print(p.get_ssn())     # ***-**-6789

p.name = "Robert"      # OK — public
p._age = 35            # Works, but against convention
```

### Example 2: Using `@property` for controlled access

```python
class Product:
    def __init__(self, name, price):
        self.name = name
        self._price = 0
        self.price = price  # Uses the setter

    @property
    def price(self):
        """Getter — returns the price."""
        return self._price

    @price.setter
    def price(self, value):
        """Setter — validates and sets the price."""
        if value < 0:
            raise ValueError("Price cannot be negative")
        self._price = value

    @property
    def tax(self):
        """Computed property (no setter — read-only)."""
        return self.price * 0.08

p = Product("Laptop", 1000)
print(p.price)    # 1000
print(p.tax)      # 80.0

p.price = 1200   # Uses setter
# p.price = -5   # ValueError!

# p.tax = 100    # AttributeError — no setter
```

### Example 3: Property with deletion

```python
class User:
    def __init__(self, username, email):
        self.username = username
        self._email = email

    @property
    def email(self):
        return self._email

    @email.setter
    def email(self, value):
        if "@" not in value:
            raise ValueError("Invalid email")
        self._email = value

    @email.deleter
    def email(self):
        print(f"Deleting email for {self.username}")
        self._email = None

u = User("alice99", "alice@example.com")
print(u.email)         # alice@example.com
u.email = "alice@new.com"
del u.email            # Deleting email for alice99
print(u.email)         # None
```

### Example 4: Why encapsulation matters — banking example

```python
class BankAccount:
    def __init__(self, owner, initial_balance=0):
        self.owner = owner
        self.__balance = initial_balance
        self.__transaction_log = []

    @property
    def balance(self):
        return self.__balance

    def deposit(self, amount):
        if amount <= 0:
            raise ValueError("Deposit must be positive")
        self.__balance += amount
        self.__log_transaction("deposit", amount)

    def withdraw(self, amount):
        if amount <= 0:
            raise ValueError("Withdrawal must be positive")
        if amount > self.__balance:
            raise ValueError("Insufficient funds")
        self.__balance -= amount
        self.__log_transaction("withdraw", amount)

    def __log_transaction(self, transaction_type, amount):
        """Private method — internal logging."""
        self.__transaction_log.append(
            f"{transaction_type}: ${amount}"
        )

    def get_transaction_history(self):
        return list(self.__transaction_log)  # Return a copy

acc = BankAccount("Alice", 1000)
acc.deposit(500)
acc.withdraw(200)
print(acc.balance)              # 1300

# Can't directly modify balance
# acc.__balance = 999999        # AttributeError
# Can't access transaction_log
# print(acc.__transaction_log)  # AttributeError

print(acc.get_transaction_history())
# ['deposit: $500', 'withdraw: $200']
```

### Example 5: Computed properties

```python
class Rectangle:
    def __init__(self, width, height):
        self.width = width
        self.height = height

    @property
    def area(self):
        return self.width * self.height

    @property
    def perimeter(self):
        return 2 * (self.width + self.height)

r = Rectangle(5, 3)
print(r.area)       # 15 — computed, not stored
print(r.perimeter)  # 16

r.width = 10
print(r.area)       # 30 — re-computed automatically
```

## Common mistakes

1. **Assuming `__private` is truly private** — name mangling is not security; the attribute is still accessible via `_ClassName__attribute`
2. **Overusing getters/setters directly** instead of using `@property` — leads to Java-style boilerplate in Python
3. **Writing getters/setters for every attribute** — only use properties when you need controlled access; start with plain attributes
4. **Forgetting that `@property` getters should have no side effects** — a getter shouldn't modify state
5. **Using `__` (double underscore) when you mean `_` (single underscore)** — `__` is for name mangling (mainly for inheritance); `_` is for "internal use"
6. **Thinking encapsulation is about security** — it's about design, not security. Python trusts its developers

## Interview questions + answers

**Q1: What is encapsulation in Python?**
A: Encapsulation bundles data and methods within a class and restricts direct access to some components. Python uses naming conventions (`_` for protected, `__` for private) and `@property` for controlled access.

**Q2: How do you create private attributes in Python?**
A: Prefix the attribute name with double underscores: `self.__attribute`. Python performs name mangling, renaming it to `_ClassName__attribute`.

**Q3: What is the difference between `_protected` and `__private`?**
A: `_protected` is a convention indicating internal use (no name change). `__private` triggers name mangling to `_ClassName__private`, primarily to avoid name collisions in inheritance.

**Q4: What is the `@property` decorator?**
A: `@property` allows defining a method that can be accessed like an attribute. It provides controlled access with getter, setter, and deleter capabilities.

**Q5: Why is encapsulation important?**
A: It protects data integrity, hides implementation details, allows validation during modification, and makes code more maintainable by separating interface from implementation.

## Revision notes (bullet points summary)

- `name` — public, accessible everywhere
- `_name` — protected, internal use convention
- `__name` — private, name mangled to `_Class__name`
- Name mangling prevents accidental overriding in subclasses, not security
- `@property` creates getter; `@prop.setter` creates setter; `@prop.deleter` creates deleter
- Properties allow validation, computed values, and read-only access
- Encapsulation protects invariants and hides implementation
- Python trusts developers — no strict enforcement
- Use `_` for internal implementation, `__` to avoid subclass name clashes
