# 5. Polymorphism

## Definition

**Polymorphism** (Greek: "many forms") is the ability of different classes to respond to the same method call in their own way. It allows you to write code that works on objects of different types as long as they support the same interface.

```python
class Cat:
    def sound(self):
        return "Meow"

class Dog:
    def sound(self):
        return "Woof"

def make_sound(animal):
    print(animal.sound())  # Polymorphic call

make_sound(Cat())  # Meow
make_sound(Dog())  # Woof
```

## Why it exists

Polymorphism exists to:
- **Write flexible, reusable code**  -  a single function can work with any object that supports the needed methods
- **Enable duck typing**  -  focus on what objects *do*, not what they *are*
- **Support the Open/Closed Principle**  -  add new types without modifying existing code
- **Simplify code**  -  avoid massive conditional chains for type checking

## How it works

Polymorphism in Python works through **duck typing**: "If it walks like a duck and quacks like a duck, it's a duck." Python doesn't check types at compile time  -  it just tries to call the method at runtime.

```python
def describe(entity):
    # Any object with .name and .sound() works
    return f"{entity.name} says {entity.sound()}"

class Dog:
    def __init__(self, name):
        self.name = name
    def sound(self):
        return "Woof"

class Cat:
    def __init__(self, name):
        self.name = name
    def sound(self):
        return "Meow"

class Car:
    def __init__(self, name):
        self.name = name
    def sound(self):
        return "Vroom"

print(describe(Dog("Rex")))     # Rex says Woof
print(describe(Cat("Luna")))    # Luna says Meow
print(describe(Car("Tesla")))   # Tesla says Vroom
```

### Method overriding (inheritance-based polymorphism)

```python
class Shape:
    def area(self):
        return 0

class Circle(Shape):
    def __init__(self, radius):
        self.radius = radius

    def area(self):                    # Override
        return 3.14 * self.radius ** 2

class Rectangle(Shape):
    def __init__(self, w, h):
        self.w = w
        self.h = h

    def area(self):                    # Override
        return self.w * self.h

shapes = [Circle(5), Rectangle(4, 6), Shape()]
for s in shapes:
    print(s.area())  # 78.5, 24, 0
```

## Internal working

Polymorphism is runtime dynamic dispatch. Python looks up the method on the object's class at the moment of the call:

```python
import dis

def call_sound(obj):
    obj.sound()

# At bytecode level, Python does LOAD_ATTR on the actual object type
# Python doesn't know what type obj has until runtime
# It checks the object's class __dict__ for 'sound' and calls it
```

Python's `len()`, `str()`, and `+` operator all use polymorphism through **special methods** (dunder methods):

| Operation | Method called |
|-----------|---------------|
| `len(obj)` | `obj.__len__()` |
| `str(obj)` | `obj.__str__()` |
| `obj + other` | `obj.__add__(other)` |
| `obj[key]` | `obj.__getitem__(key)` |

## Real-world analogy

Think of a universal remote control with a "Play" button. When you press Play:
- On a DVD player → it plays a movie
- On a music system → it plays a song
- On a game console → it starts the game

You don't care what device it is  -  you just need it to *play*. The same interface produces different behaviors depending on the object.

## Examples (with code)

### Example 1: Duck typing in action

```python
class Pigeon:
    def fly(self):
        return "Pigeon flies away"

class Airplane:
    def fly(self):
        return "Airplane takes off"

class Superman:
    def fly(self):
        return "Superman soars through the sky"

def let_it_fly(flying_thing):
    result = flying_thing.fly()
    print(result)

let_it_fly(Pigeon())       # Pigeon flies away
let_it_fly(Airplane())     # Airplane takes off
let_it_fly(Superman())     # Superman soars through the sky
```

### Example 2: Polymorphism with built-in functions

```python
class Book:
    def __init__(self, title, author):
        self.title = title
        self.author = author

    def __len__(self):
        return len(self.title)

    def __str__(self):
        return f"'{self.title}' by {self.author}"

class Playlist:
    def __init__(self, songs):
        self.songs = songs

    def __len__(self):
        return len(self.songs)

    def __str__(self):
        return f"Playlist with {len(self.songs)} songs"

book = Book("1984", "George Orwell")
playlist = Playlist(["Song1", "Song2", "Song3", "Song4"])

print(len(book))        # 4  -  via __len__
print(len(playlist))    # 4  -  via __len__
print(str(book))        # '1984' by George Orwell  -  via __str__
print(str(playlist))    # Playlist with 4 songs  -  via __str__
```

class PDFReport:

### Example 4: Polymorphism with inheritance hierarchy

```python
class Employee:
    def __init__(self, name, salary):
        self.name = name
        self.salary = salary

    def calculate_pay(self):
        return self.salary

    def work(self):
        return f"{self.name} is working"

class Manager(Employee):
    def __init__(self, name, salary, bonus):
        super().__init__(name, salary)
        self.bonus = bonus

    def calculate_pay(self):
        return self.salary + self.bonus

    def manage(self):
        return f"{self.name} is managing the team"

class Intern(Employee):
    def __init__(self, name, hourly_rate, hours_worked):
        super().__init__(name, 0)
        self.hourly_rate = hourly_rate
        self.hours_worked = hours_worked

    def calculate_pay(self):
        return self.hourly_rate * self.hours_worked

    def learn(self):
        return f"{self.name} is learning"

employees = [
    Employee("Alice", 50000),
    Manager("Bob", 80000, 15000),
    Intern("Charlie", 20, 120)
]

for emp in employees:
    print(f"{emp.name}: ${emp.calculate_pay():,.2f}")
    print(f"  {emp.work()}")
# Alice: $50,000.00
#   Alice is working
# Bob: $95,000.00
#   Bob is working
# Charlie: $2,400.00
#   Charlie is working
```

### Example 5: Operator overloading (polymorphic `+`)

```python
class Vector:
    def __init__(self, x, y):
        self.x = x
        self.y = y

    def __add__(self, other):
        return Vector(self.x + other.x, self.y + other.y)

    def __repr__(self):
        return f"Vector({self.x}, {self.y})"

v1 = Vector(1, 2)
v2 = Vector(3, 4)
v3 = v1 + v2  # Calls v1.__add__(v2)
print(v3)     # Vector(4, 6)

# Same + operator works differently for different types
print(2 + 3)           # 5  -  int addition
print("Hello " + "World")  # Hello World  -  string concatenation
print([1, 2] + [3, 4])     # [1, 2, 3, 4]  -  list concatenation
```

## Common mistakes

1. **Using `type()` or `isinstance()` checks when polymorphism should be used**  -  defeats the purpose
   ```python
   # Bad: type checking
   def process(animal):
       if type(animal) == Dog: print("Woof")
       elif type(animal) == Cat: print("Meow")

   # Good: polymorphic
   def process(animal):
       print(animal.sound())
   ```

2. **Not implementing the full interface**  -  if a function expects `.speak()` and your object doesn't have it, you get `AttributeError`

3. **Confusing polymorphism with method overloading**  -  Python doesn't have method overloading (same name, different parameters); it has polymorphism (different classes, same method signature)

4. **Forgetting that duck typing is about behavior, not inheritance**  -  the object doesn't need to share a parent class; it just needs the method

5. **Assuming polymorphism only works with inheritance**  -  duck typing is the true Pythonic way

## Interview questions + answers

**Q1: What is polymorphism?**
A: Polymorphism means "many forms"  -  the ability of different object types to respond to the same method call with their own implementation.

**Q2: What is duck typing?**
A: "If it walks like a duck and quacks like a duck, it's a duck." Python focuses on what an object can do (its methods), not what class it is. If an object has the required method, it's accepted.

**Q3: How does polymorphism differ from method overriding?**
A: Method overriding is a mechanism (child redefines a parent method). Polymorphism is the broader concept that allows different objects to respond differently to the same method call  -  method overriding enables polymorphism in inheritance hierarchies.

**Q4: Does Python have method overloading?**
A: No  -  Python doesn't support multiple methods with the same name but different parameters. You use default arguments, `*args`, or `@classmethod` alternatives instead.

**Q5: What's the difference between `isinstance()` and duck typing?**
A: `isinstance()` checks the type hierarchy. Duck typing ignores types and just tries to call methods. Idiomatic Python prefers duck typing.
