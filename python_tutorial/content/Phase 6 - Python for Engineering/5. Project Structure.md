# Project Structure

## Definition

**Project structure** is the way you organize files and folders in a Python project. A well-structured project separates concerns, makes code reusable, simplifies testing, and helps other developers (and your future self) understand the codebase quickly.

A standard Python project looks like this:

```
my_project/
├── src/
│   ├── __init__.py
│   ├── main.py
│   ├── config.py
│   ├── models.py
│   ├── services.py
│   └── utils.py
├── tests/
│   ├── __init__.py
│   ├── test_models.py
│   └── test_services.py
├── docs/
├── scripts/
├── .env
├── .gitignore
├── README.md
├── requirements.txt
└── pyproject.toml
```

## Why it exists

- **Maintainability**: When your project has 20+ files, a flat "everything in one folder" structure becomes impossible to navigate.
- **Separation of concerns**: Different responsibilities (config, business logic, data models, utilities) live in separate files. You know where to look for each thing.
- **Reusability**: Well-structured code is easier to extract into separate packages and reuse across projects.
- **Testability**: Isolated modules are easy to test. If your entire app is one 2000-line script, testing is a nightmare.
- **Team collaboration**: Multiple developers can work on different modules without merge conflicts.
- **Scalability**: A good structure scales from a small script to a large application without requiring a complete rewrite.
- **DRY principle** (Don't Repeat Yourself): Shared utilities go in one place, not copied across files.

## How it works

**The standard structure separates code by responsibility:**

| Directory/File | Purpose |
|----------------|---------|
| `src/` or `my_package/` | Main application code (as a Python package) |
| `src/__init__.py` | Makes the directory a Python package; controls exports |
| `src/main.py` | Entry point (CLI, server startup, etc.) |
| `src/config.py` | Configuration (env vars, settings classes) |
| `src/models.py` | Data models / schemas (dataclasses, ORM models, Pydantic) |
| `src/services.py` | Business logic (functions that do the work) |
| `src/utils.py` | Shared utility functions |
| `tests/` | Test files mirroring the src structure |
| `docs/` | Documentation |
| `scripts/` | Helper scripts (data migration, setup, etc.) |
| `.env` | Local environment variables (not committed) |
| `pyproject.toml` | Project metadata, dependencies, build config |
| `requirements.txt` | List of dependencies (pin versions) |
| `.gitignore` | Files git should ignore |

**How Python finds your code:**
- Python adds the project root to `sys.path`.
- `from my_package import module` looks for `my_package/__init__.py`.
- `from my_package.subpackage import thing` looks for `my_package/subpackage/__init__.py`.
- `__init__.py` can export names via `__all__ = ["func1", "ClassA"]`.

## Internal working

**How imports work in a structured project:**

When you run `python src/main.py`, Python adds the directory containing `main.py` (`src/`) to `sys.path`. But other directories (`models.py`, `services.py`) are in `src/` too, so `from models import User` works.

When you install your package (`pip install -e .`), setuptools creates a proper link so `from my_project.models import User` works from anywhere.

**How `__init__.py` works:**

```python
# src/__init__.py
from .models import User, Task
from .services import process_user, create_task
from .config import get_settings

__all__ = ["User", "Task", "process_user", "create_task", "get_settings"]
```

Now `from src import User` works, and `from src import *` imports only the names in `__all__`.

**How configuration management works:**

A `config.py` file centralizes all configuration in one place. Every other module imports config from here  -  never reads `os.environ` directly:

```python
# src/config.py
import os
from dataclasses import dataclass

@dataclass
class Settings:
    database_url: str
    api_key: str
    debug: bool
    port: int
    
    @classmethod
    def from_env(cls):
        return cls(
            database_url=os.getenv("DATABASE_URL", "sqlite:///app.db"),
            api_key=os.getenv("API_KEY", ""),
            debug=os.getenv("DEBUG", "false").lower() == "true",
            port=int(os.getenv("PORT", "8000")),
        )

settings = Settings.from_env()
```

Other files: `from src.config import settings`

## Real-world analogy

A well-structured Python project is like a **well-organized toolbox**:

- **Flat script approach**: All your tools (wrench, screwdriver, hammer, nails, tape measure) are thrown into one big box. Every time you need a screwdriver, you dig through the pile.
- **Structured approach**: The toolbox has compartments. Top tray: screwdrivers. Left compartment: wrenches. Bottom drawer: nails and screws. Side pouch: tape measure.

| Toolbox         | Python project          |
|-----------------|-------------------------|
| Screwdrivers    | `services/` (business logic) |
| Wrenches        | `models/` (data models) |
| Nails/Screws    | `config/` (settings)    |
| Tape measure    | `utils/` (shared tools) |
| Toolbox itself  | `src/` (the package)    |
| Inventory list  | `__init__.py` + `__all__` |

When you need to fix something (add a feature), you know exactly which compartment to open (which file to edit). You don't empty the entire toolbox.

### 2. `__init__.py`  -  package exports

```python
# src/__init__.py
"""Weather App  -  fetch and display weather data."""

from .config import Settings
from .fetcher import fetch_weather
from .parser import parse_weather_response
from .display import format_weather_output

__version__ = "1.0.0"
__all__ = [
    "Settings",
    "fetch_weather",
    "parse_weather_response",
    "format_weather_output",
]
```

### 3. `config.py`  -  centralised configuration

```python
# src/config.py
import os
from dataclasses import dataclass

@dataclass
class Settings:
    api_key: str
    city: str
    units: str
    timeout: int

    @classmethod
    def from_env(cls):
        return cls(
            api_key=os.getenv("WEATHER_API_KEY", ""),
            city=os.getenv("WEATHER_CITY", "London"),
            units=os.getenv("WEATHER_UNITS", "metric"),
            timeout=int(os.getenv("HTTP_TIMEOUT", "10")),
        )

settings = Settings.from_env()
```

### 4. `fetcher.py`  -  single responsibility

```python
# src/fetcher.py
import requests
from src.config import settings

def fetch_weather() -> dict:
    """Fetch weather data from OpenWeatherMap API."""
    url = "https://api.openweathermap.org/data/2.5/weather"
    params = {
        "q": settings.city,
        "appid": settings.api_key,
        "units": settings.units,
    }

    response = requests.get(url, params=params, timeout=settings.timeout)
    response.raise_for_status()
    return response.json()
```

### 5. `parser.py`  -  single responsibility

```python
# src/parser.py
from dataclasses import dataclass

@dataclass
class WeatherData:
    city: str
    temperature: float
    condition: str
    humidity: int
    wind_speed: float

def parse_weather_response(raw: dict) -> WeatherData:
    """Parse raw API response into a WeatherData object."""
    return WeatherData(
        city=raw["name"],
        temperature=raw["main"]["temp"],
        condition=raw["weather"][0]["description"],
        humidity=raw["main"]["humidity"],
        wind_speed=raw["wind"]["speed"],
    )
```

### 6. `display.py`  -  single responsibility

```python
# src/display.py
from src.parser import WeatherData

def format_weather_output(data: WeatherData) -> str:
    """Format weather data as a human-readable string."""
    return (
        f"Weather in {data.city}:\n"
        f"    {data.temperature:.1f}°C\n"
        f"    {data.condition}\n"
        f"   {data.humidity}%\n"
        f"   {data.wind_speed} m/s"
    )
```

### 7. `main.py`  -  entry point

```python
# src/main.py
from src.fetcher import fetch_weather
from src.parser import parse_weather_response
from src.display import format_weather_output

def main():
    try:
        raw = fetch_weather()
        data = parse_weather_response(raw)
        output = format_weather_output(data)
        print(output)
    except Exception as e:
        print(f"Error: {e}")

if __name__ == "__main__":
    main()
```

### 8. `tests/` mirroring structure

```python
# tests/test_parser.py
from src.parser import parse_weather_response

def test_parse_weather_response():
    raw = {
        "name": "London",
        "main": {"temp": 15.5, "humidity": 72},
        "weather": [{"description": "clear sky"}],
        "wind": {"speed": 3.6},
    }
    data = parse_weather_response(raw)
    assert data.city == "London"
    assert data.temperature == 15.5
    assert data.humidity == 72
```

**Benefits of restructuring:**
- Each file has one responsibility (Single Responsibility Principle).
- Tests can import and test each module independently.
- Adding a new data source (e.g., forecast) means adding a new file, not bloating `main.py`.
- Configuration is centralized  -  change one file, not 10.
- Installing with `pip install -e .` makes `weather` a globally available CLI command.

## Common mistakes

- **One giant file**: A 1000+ line `main.py` with functions, classes, global state, and spaghetti logic. Break it into modules.
- **Circular imports**: `a.py` imports from `b.py` which imports from `a.py`. Restructure to avoid this (move shared code to a third module).
- **`__init__.py` in test directories**: Not always needed (pytest handles it), but create them if imports fail.
- **Hardcoding paths**: Use `pathlib.Path(__file__).parent` for relative paths, not strings like `"../data/"`.
- **No `__init__.py` in src**: Without it, Python doesn't treat `src/` as a package, and imports fail.
- **Too many tiny files**: 100 files with 3 lines each is as bad as one giant file. Find the right balance.
- **Mixing concerns**: Don't put database logic in the same file as UI rendering. Separate data, logic, and presentation.
- **Not using `__all__`**: Without it, `from package import *` exports everything (including private functions starting with `_`).
- **Installing vs running directly**: Running `python src/main.py` works but `import src.config` might not. Use `pip install -e .` and run via entry points.

## Interview questions + answers

**Q1: Why separate code into `models.py`, `services.py`, `utils.py`?**
A: This follows the Single Responsibility Principle. Each file has one concern: models = data structure, services = business logic, utils = shared helpers. Makes the code easier to navigate, test, and maintain.

**Q2: What does `__init__.py` do?**
A: It marks a directory as a Python package (required for imports). It can also export names via `__all__`, run initialization code, or define `__version__`.

**Q3: What is `__all__` and why use it?**
A: `__all__` is a list of strings defining the public API of a module/package. It controls what `from module import *` exports and documents what's meant to be used externally.

**Q4: How do you avoid circular imports?**
A: (1) Move shared code to a third module both import from. (2) Use lazy imports (import inside the function, not at top of file). (3) Restructure so the dependency goes one way only.

**Q5: What is the DRY principle?**
A: Don't Repeat Yourself. Every piece of knowledge should have a single, unambiguous representation. Put shared code in a utility module, don't copy-paste it across files.

**Q6: What is the Single Responsibility Principle?**
A: A module/class/function should have one reason to change. A `parser.py` should only parse data, not fetch it, not display it, not save it to a database.

**Q7: How do you manage configuration in a structured project?**
A: Centralize in `config.py` or `settings.py`. Read from environment variables once and expose as a dataclass/dict. Other modules import settings from there.

**Q8: What's the difference between `src/` and putting code in the project root?**
A: Putting code in `src/` forces you to install the package (`pip install -e .`), which catches import issues early and makes the package properly installable. Root-level packages can work for simple projects but don't scale.

## Knowledge check (5-10 questions with answers)

**Q1: What file marks a directory as a Python package?**
A: `__init__.py`

**Q2: What's `__all__` used for?**
A: To define the public API of a module  -  controls what `from module import *` exports.

**Q3: Name three directories every well-structured Python project should have.**
A: `src/` (source code), `tests/` (test files), `docs/` (documentation).

**Q4: What's wrong with putting all code in `main.py`?**
A: It violates Single Responsibility, makes testing hard, and becomes unmaintainable as the project grows.

**Q5: How do you make a Python package installable in development mode?**
A: Create a `pyproject.toml` with setuptools config, then run `pip install -e .` from the project root.

**Q6: What problem does `src/` layout solve?**
A: It ensures imports work correctly when the package is installed (not just when running from the project root).

**Q7: Give an example of circular imports.**
A: `a.py` has `from b import func_b` and `b.py` has `from a import func_a`. Python can't resolve this because each file requires the other to be fully loaded first.

**Q8: What does `from src.parser import parse_weather_response` look for?**
A: It looks for `src/parser.py` (with `src/__init__.py` existing), then imports the function `parse_weather_response`.

**Q9: Where do pinned dependency versions go  -  `pyproject.toml` or `requirements.txt`?**
A: Loose version constraints in `pyproject.toml` (e.g., `requests>=2.28`). Pinned exact versions in `requirements.txt` (e.g., `requests==2.31.0`) for reproducible installs.

**Q10: What is a "flat is better than nested" trap for project structure?**
A: Over-nesting (e.g., `src/services/parsers/json/weather/`) when the project is small. Flat structures are simpler. Nest only when a directory has 5-7+ files.

## Revision notes (bullet points summary)

- Standard structure: `src/`, `tests/`, `docs/`, `pyproject.toml`, `.env`, `.gitignore`.
- `__init__.py` makes a directory a package; `__all__` defines public exports.
- Separate concerns: `models.py` (data), `services.py` (logic), `utils.py` (helpers), `config.py` (settings).
- Single Responsibility Principle: each file has one reason to change.
- DRY: put shared code in one place, don't copy-paste.
- Centralize configuration in `config.py` using dataclasses + env vars.
- Avoid circular imports by restructuring or using lazy imports.
- Install in dev mode: `pip install -e .` for proper imports and entry points.
- `pyproject.toml` for metadata and deps; `requirements.txt` for pinned versions.
- Good structure scales from 1 file to 100+ without pain.
