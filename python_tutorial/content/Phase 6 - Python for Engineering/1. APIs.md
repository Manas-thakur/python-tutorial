# APIs

## Definition

An **API (Application Programming Interface)** is a set of rules and protocols that allows one software application to communicate with another. In Python, working with **Web APIs** (typically over HTTP) lets your programs fetch data from or send data to remote servers  -  think weather data, GitHub repos, Twitter feeds, or AI models.

The most common type of web API today is a **REST API**, which uses standard HTTP methods (GET, POST, PUT, DELETE, PATCH) to perform CRUD (Create, Read, Update, Delete) operations on resources.

## Why it exists

No program is an island. Modern applications need to:

- Fetch live data (weather, stock prices, news)
- Authenticate users via third-party services (Google, GitHub)
- Delegate work to cloud services (send emails, process payments, run ML models)
- Expose their own functionality for other programs to consume

APIs provide a **language-agnostic, standardized interface**  -  a Python app, a JavaScript frontend, and a mobile app can all talk to the same backend API over HTTP. Without APIs, every integration would require custom protocols and libraries, making the internet vastly less connected.

## How it works

1. Your Python program sends an **HTTP request** to a server's URL (endpoint).
2. The server processes the request, performs the action (query database, compute something, etc.).
3. The server sends back an **HTTP response** with a status code and (usually) a body  -  most commonly JSON.
4. Your program parses the response and uses the data.

```
[Your App]  -- HTTP Request -->  [API Server] -- query --> [Database]
[Your App]  <-- HTTP Response -- [API Server] <-- result -- [Database]
```

**HTTP Methods (verbs):**

| Method  | Purpose          | CRUD equivalent | Body? | Idempotent? |
|---------|------------------|-----------------|-------|-------------|
| GET     | Retrieve data    | Read            | No    | Yes         |
| POST    | Create new resource | Create       | Yes   | No          |
| PUT     | Replace resource entirely | Update/Replace | Yes | Yes    |
| PATCH   | Partially update resource | Update  | Yes   | No          |
| DELETE  | Remove resource  | Delete          | Optional | Yes    |

**HTTP Status Codes:**

| Code | Meaning                | When you see it                                      |
|------|------------------------|------------------------------------------------------|
| 200  | OK                     | GET/PUT/PATCH/DELETE succeeded                       |
| 201  | Created                | POST succeeded (new resource created)                |
| 204  | No Content             | DELETE succeeded (no body returned)                  |
| 400  | Bad Request            | Client sent malformed data (e.g. missing required field) |
| 401  | Unauthorized           | Missing or invalid authentication                       |
| 403  | Forbidden              | Authenticated but not allowed (permissions)             |
| 404  | Not Found              | Resource doesn't exist at that URL                      |
| 429  | Too Many Requests      | Rate limited  -  slow down!                               |
| 500  | Internal Server Error  | Server-side problem (not your fault)                    |
| 502  | Bad Gateway            | Upstream server failed                                  |
| 503  | Service Unavailable    | Server overloaded or under maintenance                  |

## Internal working

When you call `requests.get("https://api.github.com/users/octocat")`, here's the stack:

1. **DNS resolution**: The `requests` library (via `urllib3`) resolves `api.github.com` to an IP address.
2. **TCP connection**: A TCP socket is opened to that IP on port 443 (HTTPS).
3. **TLS handshake**: SSL/TLS certificates are exchanged to establish an encrypted channel.
4. **HTTP request construction**: `requests` builds a raw HTTP/1.1 request string:
   ```
   GET /users/octocat HTTP/1.1
   Host: api.github.com
   User-Agent: python-requests/2.x.x
   Accept-Encoding: gzip, deflate
   Accept: */*
   ```
5. **Send**: The raw bytes are written to the socket.
6. **Receive**: The server writes response bytes back.
7. **Parse**: `requests` parses the HTTP response line, headers, and body.
8. **Decode**: If the body is JSON, `response.json()` decodes it into Python dicts/lists.
9. **Connection pooling**: `urllib3` keeps the TCP connection alive (keep-alive) for reuse via `Session` objects.

## Real-world analogy

An API is like a **restaurant**:

- **You (client)** want food (data/functionality).
- **The menu** is the API documentation  -  it tells you what's available and how to order.
- **The waiter** is the API endpoint  -  you tell the waiter what you want.
- **The kitchen** is the server  -  where the actual work happens.
- **The plate they bring back** is the HTTP response (with your data).
- **HTTP methods**: Looking at the menu = GET, ordering a new dish = POST, sending back a dish for replacement = PUT, asking for a slight change = PATCH, sending back an empty plate = DELETE.
- **Status codes**: "We have that" (200), "We made it fresh" (201), "That's not on the menu" (404), "You need to be seated first" (401), "You're ordering too fast, slow down" (429).
- **Authentication** = showing your reservation confirmation.

## Examples (with code)

### 1. Basic GET request

```python
import requests

response = requests.get("https://api.github.com")
print(response.status_code)  # 200
print(response.json())       # dict of API info
```

### 2. GET with query parameters

```python
import requests

# Search GitHub repos
params = {"q": "python", "sort": "stars", "per_page": 5}
response = requests.get("https://api.github.com/search/repositories", params=params)

if response.status_code == 200:
    data = response.json()
    for repo in data["items"]:
        print(f"{repo['full_name']} - {repo['stargazers_count']} stars")
else:
    print(f"Error: {response.status_code}")
```

### 3. POST with JSON body

```python
import requests

new_post = {"title": "Hello API", "body": "Learning requests!", "userId": 1}
response = requests.post("https://jsonplaceholder.typicode.com/posts", json=new_post)

if response.status_code == 201:
    created = response.json()
    print(f"Created post #{created['id']}")
```

### 4. PUT (replace entire resource)

```python
response = requests.put(
    "https://jsonplaceholder.typicode.com/posts/1",
    json={"id": 1, "title": "Updated", "body": "New body", "userId": 1}
)
print(response.status_code)  # 200
```

### 5. PATCH (partial update)

```python
response = requests.patch(
    "https://jsonplaceholder.typicode.com/posts/1",
    json={"title": "Only title changed"}
)
print(response.status_code)  # 200
```

### 6. DELETE

```python
response = requests.delete("https://jsonplaceholder.typicode.com/posts/1")
print(response.status_code)  # 200 (or 204)
```

### 7. Headers

```python
headers = {
    "Authorization": "Bearer YOUR_TOKEN",
    "Accept": "application/vnd.github.v3+json",
    "User-Agent": "MyApp/1.0"
}
response = requests.get("https://api.github.com/user", headers=headers)
```

### 8. Authentication methods

```python
import requests
from requests.auth import HTTPBasicAuth

# Basic Auth
response = requests.get(
    "https://api.github.com/user",
    auth=HTTPBasicAuth("username", "password_or_token")
)

# API key via header
headers = {"X-API-Key": "your-api-key"}
response = requests.get("https://api.example.com/data", headers=headers)

# Bearer token (most common)
headers = {"Authorization": "Bearer YOUR_TOKEN"}
response = requests.get("https://api.example.com/protected", headers=headers)
```

### 9. Session objects (connection reuse)

```python
import requests

session = requests.Session()
session.headers.update({"Authorization": "Bearer YOUR_TOKEN", "User-Agent": "MyApp/1.0"})

# Reuse session for multiple requests  -  faster (connection pooling)
user = session.get("https://api.github.com/user").json()
repos = session.get("https://api.github.com/user/repos").json()
```

### 10. Timeouts and error handling

```python
import requests
from requests.exceptions import Timeout, ConnectionError, HTTPError

try:
    response = requests.get(
        "https://api.github.com/users/octocat",
        timeout=5          # seconds; raises Timeout if server doesn't respond
    )
    response.raise_for_status()  # raises HTTPError for 4xx/5xx
    data = response.json()
    print(data["login"])

except Timeout:
    print("Request timed out  -  server too slow or unreachable")
except ConnectionError:
    print("Could not connect  -  check your internet or the URL")
except HTTPError as e:
    print(f"HTTP error: {e.response.status_code}  -  {e.response.reason}")
except ValueError:
    print("Response wasn't valid JSON")
```

### 11. Rate limiting  -  check before hitting the API

```python
import requests
import time

response = requests.get("https://api.github.com/rate_limit")
rate_data = response.json()

core = rate_data["resources"]["core"]
print(f"Remaining: {core['remaining']} / {core['limit']}")
print(f"Resets at: {core['reset']}")

if core["remaining"] == 0:
    wait_time = core["reset"] - time.time()
    print(f"Rate limited! Wait {wait_time:.0f} seconds")
    time.sleep(max(wait_time, 0) + 1)
```

**Rate-limiting best practices:**
- Always check rate limit headers (`X-RateLimit-Remaining`, `X-RateLimit-Reset`)
- Use exponential backoff when you get a 429 response
- Cache responses when possible (don't re-fetch data you already have)
- Add delays between requests (`time.sleep(1)`) for high-frequency calls
- Use a session to avoid reconnecting for every request
- Respect `Retry-After` header if the server sends it

## Common mistakes

- **Forgetting `.json()`**: `response` is a Response object, not the data. Always call `.json()`.
- **Hardcoding API keys/URLs**: Use environment variables (see next file).
- **Not handling errors**: `response.ok` or `response.raise_for_status()`  -  never assume success.
- **No timeout**: A request can hang forever. Always set `timeout`.
- **Wrong HTTP method**: Using GET to create data or POST to fetch data.
- **Ignoring pagination**: Many APIs return only 30 items at a time. Check `Link` headers or `next` keys.
- **Over-fetching in loops**: Don't call the API 100 times in a loop  -  use batch endpoints if available.
- **Sending form data instead of JSON**: Use `json=` parameter, not `data=`, unless the API expects form-encoded data.

## Interview questions + answers

**Q1: What's the difference between PUT and PATCH?**
A: PUT replaces the entire resource. PATCH applies a partial update. If you PUT and omit a field, that field is deleted/set to default. With PATCH, only the fields you include are changed.

**Q2: What does idempotent mean, and which HTTP methods are idempotent?**
A: Idempotent means making the same request multiple times has the same effect as making it once. GET, PUT, DELETE, HEAD, OPTIONS are idempotent. POST and PATCH are not (calling POST twice creates two resources).

**Q3: Explain REST principles.**
A: REST (Representational State Transfer) requires: (1) Stateless  -  each request contains all info needed; no server-side session. (2) Resource-based  -  URLs represent resources (`/users/42`), not actions. (3) Standard HTTP methods for CRUD. (4) Representations (JSON, XML) decouple client from server internals.

**Q4: What's the difference between 401 and 403?**
A: 401 Unauthorized means "you haven't authenticated" (no token or invalid credentials). 403 Forbidden means "you're authenticated but not allowed" (insufficient permissions).

**Q5: How would you handle pagination in a REST API?**
A: Check the API docs for pagination method. Common approaches: (1) Link headers with `rel="next"`, (2) `page`/`per_page` query params, (3) cursor-based pagination with a `next` key in the response. Loop until no more pages.

**Q6: What does `response.raise_for_status()` do?**
A: It checks if the HTTP status code indicates an error (4xx or 5xx) and raises `requests.exceptions.HTTPError` if so. It's a concise way to handle bad responses.

**Q7: Why use a Session object instead of raw requests?**
A: Sessions persist cookies, reuse TCP connections (connection pooling), and allow setting default headers/auth once. This is faster and more efficient for multiple requests to the same server.

**Q8: What is rate limiting and how do you handle it?**
A: Rate limiting is when an API restricts the number of requests you can make in a time window. Handle it by: checking remaining limit headers, implementing exponential backoff on 429, caching responses, and adding delays between requests.

## Knowledge check (5-10 questions with answers)

**Q1: Which HTTP method would you use to retrieve a list of users?**
A: GET

**Q2: What status code indicates a resource was successfully created via POST?**
A: 201

**Q3: Write a one-liner to send a POST request with JSON `{"name": "Alice"}` to `https://api.example.com/users`.**
A: `requests.post("https://api.example.com/users", json={"name": "Alice"})`

**Q4: How do you access the actual data from a Response object?**
A: Call `.json()` (for JSON responses) or `.text` (for raw text).

**Q5: What exception does `raise_for_status()` raise for a 404 response?**
A: `requests.exceptions.HTTPError`

**Q6: If you see status code 429, what should your code do?**
A: Wait and retry after the time specified in the `Retry-After` header, or implement exponential backoff.

**Q7: What's wrong with `requests.get(url, timeout=None)`?**
A: It can hang forever if the server doesn't respond. Always set a numeric timeout (e.g. 5 or 10 seconds).

**Q8: Name two ways to pass authentication in a request.**
A: (1) `auth=HTTPBasicAuth(user, pass)` (2) `headers={"Authorization": "Bearer TOKEN"}`

**Q9: Is POST idempotent? Why does it matter?**
A: No. Calling POST twice creates two resources. This matters for retry logic  -  you can't safely retry a POST without knowing if it succeeded.

**Q10: What does the `params` argument do in `requests.get()`?**
A: It adds URL query parameters. `requests.get(url, params={"q": "python"})` requests `url?q=python`.

## Revision notes (bullet points summary)

- APIs let programs talk to each other over HTTP.
- GET = read, POST = create, PUT = replace, PATCH = update, DELETE = remove.
- 200 OK, 201 Created, 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 429 Too Many Requests, 500 Server Error.
- REST is stateless, resource-based, uses standard HTTP methods.
- `requests.get(url)`, `.post(url, json=...)`, `.put(url, json=...)`, `.delete(url)`.
- Always call `.json()` to get data; use `raise_for_status()` for error handling.
- Always set `timeout` to prevent hanging requests.
- Use `Session` for persistent connections with headers/auth.
- Rate limiting: check headers, implement backoff, cache, add delays.
- Never hardcode keys  -  use environment variables.
