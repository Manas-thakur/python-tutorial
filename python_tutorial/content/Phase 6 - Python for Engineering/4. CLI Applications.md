# CLI Applications

## Definition

A **CLI (Command-Line Interface) application** is a program that you run from a terminal, passing arguments and flags to control its behaviour. Examples: `git`, `docker`, `pip`, `ls`, `grep`.

In Python, you build CLI apps using:
- **`sys.argv`**  -  the raw list of command-line arguments (too basic for real projects).
- **`argparse`**  -  built-in module for parsing arguments with help messages, type checking, and defaults (standard choice).
- **`typer`**  -  modern third-party library that uses type hints and decorators (simpler, more Pythonic).

## Why it exists

- **Automation**: CLI tools are scriptable. You can chain them in shell scripts, run them in cron jobs, or trigger them from CI/CD pipelines.
- **Server-side**: Many servers don't have a GUI. CLI tools are the primary way to interact with remote machines.
- **Simplicity**: A well-designed CLI tool is often faster to use than a GUI for power users.
- **Interoperability**: Any programming language can invoke a CLI tool. It's the universal interface.
- **Developer tools**: Linters, formatters, test runners, build tools  -  all are CLI applications.

## How it works

When you run `python myapp.py --name Alice --verbose`, the shell passes the arguments to Python as a list of strings in `sys.argv`:

```python
import sys
print(sys.argv)
# ['myapp.py', '--name', 'Alice', '--verbose']
```

`argparse` parses this list into a structured object:

```python
import argparse

parser = argparse.ArgumentParser(description="My CLI tool")
parser.add_argument("--name", help="Your name")
parser.add_argument("--verbose", action="store_true", help="Enable verbose output")

args = parser.parse_args()
print(args.name)      # "Alice"
print(args.verbose)   # True
```

**Argument types:**
- **Positional args**: Required, no dashes. Order matters. `parser.add_argument("filename")` → `python app.py data.csv`
- **Optional args**: Start with `-` or `--`. `parser.add_argument("--output")` → `python app.py --output result.txt`
- **Flags**: `action="store_true"` or `"store_false"`. `parser.add_argument("--verbose", action="store_true")` → `python app.py --verbose`

## Internal working

**How `argparse` works step by step:**

1. `ArgumentParser()` creates a parser object with metadata (description, epilog, etc.).
2. `add_argument()` registers each expected argument with its configuration: name, type, default, help, choices, action, nargs, etc.
3. `parse_args()` calls `sys.argv` (or an explicit list), then:
   - Tokenizes the arguments into a list of strings.
   - Matches optional args by looking for `-`/`--` prefixes.
   - Matches positional args by order (after optional args are consumed).
   - Converts each value using the specified `type` function (e.g., `int`, `float`, `open`).
   - Runs validation (`choices`, `required`, range checks).
   - Returns a `Namespace` object where each argument is an attribute.
4. If `-h`/`--help` is encountered, the parser prints a formatted help message and exits.

**How entry points work (making `python myapp.py` into `myapp`):**

With `pyproject.toml` or `setup.py`, you define a console script entry point. When the package is installed, pip creates a small executable wrapper that calls your Python function:

```toml
# pyproject.toml
[project.scripts]
myapp = "myapp.cli:main"
```

This is equivalent to creating a shell script like:
```bash
#!/usr/bin/env python
from myapp.cli import main
main()
```

Python's `entry_points` mechanism uses `setuptools` to generate these wrapper scripts during `pip install`.

## Real-world analogy

A CLI application is like a **remote control**:

- **The program** = the TV.
- **Arguments** = buttons on the remote. Each button does something specific.
- **Positional arguments** = the channel number. You always need one, and it goes first.
- **Optional arguments** = volume up/down. You use them when you want, and they have sensible defaults (volume = 20).
- **Flags** = the power button or mute. They don't take a value  -  they just toggle a state.
- **`--help`** = the "Menu" button that shows what all buttons do.
- **Subparsers** = the TV + DVD + Soundbar remotes combined into one. First you pick which device, then you send device-specific commands.

### 3. argparse  -  positional + optional + flag

```python
# file_processor.py
import argparse

parser = argparse.ArgumentParser(description="Process a file")

# Positional
parser.add_argument("input_file", help="Input file path")

# Optional with type
parser.add_argument("--output", "-o", default="output.txt", help="Output file path")
parser.add_argument("--count", "-c", type=int, default=10, help="Number of lines")
parser.add_argument("--encoding", choices=["utf-8", "latin-1"], default="utf-8")

# Flag
parser.add_argument("--verbose", "-v", action="store_true", help="Enable verbose output")
parser.add_argument("--quiet", "-q", action="store_true", help="Suppress output")

args = parser.parse_args()

if args.verbose:
    print(f"Processing {args.input_file}...")
    print(f"Output: {args.output}, Count: {args.count}")
    print(f"Encoding: {args.encoding}")

# ... process the file ...
```

```bash
$ python file_processor.py data.csv -o result.csv -c 20 --verbose
Processing data.csv...
Output: result.csv, Count: 20
Encoding: utf-8
```

### 4. argparse with type checking and defaults

```python
import argparse

def valid_port(value):
    """Custom type checker."""
    port = int(value)
    if not (1 <= port <= 65535):
        raise argparse.ArgumentTypeError(f"Invalid port: {port}")
    return port

parser = argparse.ArgumentParser(description="Start a server")
parser.add_argument("--host", default="127.0.0.1", help="Host address")
parser.add_argument("--port", type=valid_port, default=8080, help="Port number (1-65535)")
parser.add_argument("--workers", type=int, choices=[1, 2, 4, 8], default=4)
parser.add_argument("--debug", action="store_true")

args = parser.parse_args()
print(f"Starting on {args.host}:{args.port} with {args.workers} workers")
```

```bash
$ python server.py --port 99999
usage: server.py [-h] [--host HOST] [--port PORT] [--workers {1,2,4,8}] [--debug]
server.py: error: argument --port: Invalid port: 99999
```

### 5. Subparsers (complex CLI tools)

```python
# todo_cli.py
import argparse

def main():
    parser = argparse.ArgumentParser(description="CLI To-Do App")
    subparsers = parser.add_subparsers(dest="command", help="Available commands")

    # Add command
    add_parser = subparsers.add_parser("add", help="Add a new task")
    add_parser.add_argument("task", help="Task description")
    add_parser.add_argument("--priority", "-p", choices=["low", "medium", "high"], default="medium")

    # List command
    list_parser = subparsers.add_parser("list", help="List all tasks")
    list_parser.add_argument("--all", action="store_true", help="Show completed too")

    # Done command
    done_parser = subparsers.add_parser("done", help="Mark task as done")
    done_parser.add_argument("task_id", type=int, help="Task ID to mark done")

    args = parser.parse_args()

    if args.command == "add":
        print(f"Adding task: {args.task} (priority: {args.priority})")
    elif args.command == "list":
        mode = "all" if args.all else "pending"
        print(f"Listing tasks: {mode}")
    elif args.command == "done":
        print(f"Marking task #{args.task_id} as done")
    else:
        parser.print_help()

if __name__ == "__main__":
    main()
```

```bash
$ python todo_cli.py add "Buy groceries" -p high
Adding task: Buy groceries (priority: high)

$ python todo_cli.py list
Listing tasks: pending

$ python todo_cli.py done 1
Marking task #1 as done
```

### 7. Typer library (modern alternative)

First install: `pip install typer`

```python
# hello_typer.py
import typer

app = typer.Typer()

@app.command()
def greet(name: str, count: int = 1, uppercase: bool = False):
    """Greet someone with style."""
    message = f"Hello, {name}!"
    if uppercase:
        message = message.upper()
    
    for _ in range(count):
        print(message)

@app.command()
def goodbye(name: str, formal: bool = False):
    """Say goodbye."""
    if formal:
        print(f"Goodbye, {name}. It was a pleasure.")
    else:
        print(f"See ya, {name}!")

if __name__ == "__main__":
    app()
```

```bash
$ python hello_typer.py greet Alice --count 3 --uppercase
HELLO, ALICE!
HELLO, ALICE!
HELLO, ALICE!

$ python hello_typer.py goodbye Bob
See ya, Bob!
```

Typer automatically generates help, validates types, and handles subcommands  -  all from type hints.

### Creative mini project: CLI To-Do app with argparse

```python
# todo.py
import argparse
import json
import os

DATA_FILE = "tasks.json"

def load_tasks():
    if not os.path.exists(DATA_FILE):
        return []
    with open(DATA_FILE, "r") as f:
        return json.load(f)

def save_tasks(tasks):
    with open(DATA_FILE, "w") as f:
        json.dump(tasks, f, indent=2)

def main():
    parser = argparse.ArgumentParser(description="Simple CLI To-Do App")
    subparsers = parser.add_subparsers(dest="command", help="Commands")

    # add
    add_parser = subparsers.add_parser("add", help="Add a new task")
    add_parser.add_argument("task", help="Task description")
    add_parser.add_argument("--priority", "-p", choices=["low", "medium", "high"], default="medium")

    # list
    subparsers.add_parser("list", help="List pending tasks")

    # done
    done_parser = subparsers.add_parser("done", help="Mark task as done")
    done_parser.add_argument("task_id", type=int, help="Task number to mark done")

    # delete
    delete_parser = subparsers.add_parser("delete", help="Delete a task")
    delete_parser.add_argument("task_id", type=int, help="Task number to delete")

    # clear
    subparsers.add_parser("clear", help="Delete all tasks")

    args = parser.parse_args()

    if args.command == "add":
        tasks = load_tasks()
        task = {
            "id": len(tasks) + 1,
            "task": args.task,
            "priority": args.priority,
            "done": False,
        }
        tasks.append(task)
        save_tasks(tasks)
        print(f"Added task #{task['id']}: {args.task} [{args.priority}]")

    elif args.command == "list":
        tasks = load_tasks()
        if not tasks:
            print("No tasks!")
            return
        for t in tasks:
            status = "[x]" if t["done"] else "[ ]"
            print(f"{t['id']}. {status} {t['task']} ({t['priority']})")

    elif args.command == "done":
        tasks = load_tasks()
        found = False
        for t in tasks:
            if t["id"] == args.task_id:
                t["done"] = True
                found = True
                break
        if found:
            save_tasks(tasks)
            print(f"Task #{args.task_id} marked as done!")
        else:
            print(f"Task #{args.task_id} not found!")

    elif args.command == "delete":
        tasks = load_tasks()
        tasks = [t for t in tasks if t["id"] != args.task_id]
        save_tasks(tasks)
        print(f"Task #{args.task_id} deleted!")

    elif args.command == "clear":
        save_tasks([])
        print("All tasks cleared!")

    else:
        parser.print_help()

if __name__ == "__main__":
    main()
```

```bash
$ python todo.py add "Buy milk"
Added task #1: Buy milk [medium]
$ python todo.py add "Write report" -p high
Added task #2: Write report [high]
$ python todo.py add "Call mom" -p low
Added task #3: Call mom [low]
$ python todo.py list
1. [ ] Buy milk (medium)
2. [ ] Write report (high)
3. [ ] Call mom (low)
$ python todo.py done 1
Task #1 marked as done!
$ python todo.py list
1. [x] Buy milk (medium)
2. [ ] Write report (high)
3. [ ] Call mom (low)
$ python todo.py delete 3
Task #3 deleted!
```

## Common mistakes

- **Forgetting to call `parse_args()`**: Without it, no arguments are parsed.
- **Using `action="store"` for flags**: Flags should use `action="store_true"` or `"store_false"`, not "store".
- **Not handling missing required args**: Positional args are required by default. Optional args with no default are `None`.
- **Confusing positional and optional**: Positional args have no dashes. Don't require the user to type `--filename` when they could just type the filename.
- **Bad help messages**: Always write clear help text. Your future self will thank you.
- **Not converting types**: `args.count` is a string unless you specify `type=int`.
- **Too many subcommands**: Don't make users type `app command subcommand action --flag`. Keep it simple.
- **Exit codes**: Return `sys.exit(1)` on errors. Pipes and scripts check exit codes.
- **Not testing the CLI**: Use `subprocess` or `CliRunner` (from Typer or Click) to test your CLI in tests.

## Interview questions + answers

**Q1: What is `sys.argv`?**
A: `sys.argv` is a list of command-line arguments passed to a Python script. `sys.argv[0]` is the script name, `sys.argv[1:]` are the actual arguments. Too basic for real tools  -  use `argparse`.

**Q2: How do you make a CLI tool installable (so `myapp` works from anywhere)?**
A: Define an entry point in `pyproject.toml` under `[project.scripts]`: `myapp = "mypackage.cli:main"`. Then `pip install` the package.

**Q3: What's the difference between positional and optional arguments in argparse?**
A: Positional args are required, have no dashes, and are identified by order. Optional args start with `-` or `--`, are... optional, and can be passed in any order.

**Q4: How do you create subcommands like `git commit` and `git push`?**
A: Use `subparsers = parser.add_subparsers(dest="command")` then `add_parser("commit")`, `add_parser("push")`, etc. Check `args.command` to dispatch.

**Q5: What does `action="store_true"` do?**
A: It stores `True` if the flag is present, `False` otherwise. Used for boolean flags like `--verbose`.

**Q6: How do you restrict an argument to specific values?**
A: Use the `choices` parameter: `parser.add_argument("--level", choices=["debug", "info", "error"])`.

**Q7: How does Typer differ from argparse?**
A: Typer uses type hints and decorators (`@app.command()`) instead of manually building a parser. It's more concise and Pythonic. Arguments become function parameters with types.

**Q8: What happens if a required positional argument is missing?**
A: `argparse` prints an error message with usage info and exits with a non-zero code.

**Q9: How do you pass a default value for an optional argument?**
A: `parser.add_argument("--output", default="output.txt")`. If not provided, `args.output` will be `"output.txt"`.

**Q10: What is `argparse.ArgumentTypeError`?**
A: It's an exception you raise in a custom type function to provide a clear error message when a value is invalid.

## Knowledge check (5-10 questions with answers)

**Q1: What module is the standard choice for building CLI apps in Python?**
A: `argparse` (built-in).

**Q2: Write the code to create an argparse parser with a required positional argument `filename`.**
A:
```python
parser = argparse.ArgumentParser()
parser.add_argument("filename")
args = parser.parse_args()
```

**Q3: How do you add a `--verbose` flag that stores `True` when present?**
A: `parser.add_argument("--verbose", action="store_true")`

**Q4: What does `sys.argv[0]` contain?**
A: The script name/path (e.g., `"myscript.py"`).

**Q5: How do you create subcommands in argparse?**
A: `subparsers = parser.add_subparsers(dest="command")`, then `subparsers.add_parser("command_name")`.

**Q6: What happens if `--port` is required but the user runs `python app.py` without it?**
A: If `--port` is optional with no default, `args.port` will be `None`. If it's marked `required=True`, argparse prints an error and exits.

**Q7: Write a one-line argument definition that accepts an integer between 1 and 10.**
A: `parser.add_argument("--level", type=int, choices=range(1, 11))`

**Q8: What's the modern third-party alternative to argparse that uses type hints?**
A: Typer.

**Q9: What is an entry point in pyproject.toml?**
A: It maps a CLI command name to a Python function: `mycmd = "mypackage.module:function"`. After `pip install`, `mycmd` runs that function.

**Q10: What does `sys.exit(1)` do in a CLI tool?**
A: Exits the program with exit code 1 (indicating an error to the shell/OS).

## Revision notes (bullet points summary)

- CLI apps accept command-line arguments to control behaviour.
- `sys.argv` gives raw args; `argparse` parses them properly.
- Positional args = required, no dashes. Optional args = `--name`, `-n`, with defaults.
- Flags (`action="store_true"`) = presence/absence of a toggle.
- `type=int`, `choices=[...]`, `default=...` control validation.
- Subparsers (`add_subparsers`) create multi-command tools (git-style).
- Entry points (`pyproject.toml` `[project.scripts]`) make tools globally callable.
- Typer (decorator-based, type-hint-driven) is a modern alternative.
- Always clear help messages (`-h`/`--help` auto-generated).
- Use `sys.exit(1)` on errors, handle missing args gracefully.
