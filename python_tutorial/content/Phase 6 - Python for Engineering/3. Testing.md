# Testing

## Definition

**Testing** is the practice of writing code that verifies your application code behaves correctly. A test runs a piece of your code with known inputs and asserts that the output or side effects match expectations.

Python has two major testing frameworks:
- **`unittest`** — built-in, class-based, inspired by JUnit (Java).
- **`pytest`** — third-party, function-based, simpler syntax, more powerful (industry standard today).

Testing is automated — you write tests once and run them thousands of times, catching regressions (bugs introduced by new code) before they reach users.

## Why it exists

- **Catch bugs early**: A bug found in production costs 10x more than one caught during development. Testing catches mistakes seconds after you make them.
- **Refactor with confidence**: Want to rewrite a function? If you have good tests, you can refactor fearlessly — the tests will tell you if you broke something.
- **Documentation**: Tests are executable documentation. They show exactly how a function is supposed to be used and what it returns.
- **Design pressure**: Writing testable code forces you to write modular, loosely coupled code with clear inputs and outputs.
- **Automation**: Tests can run automatically on every commit (CI/CD pipeline), giving you an instant safety net.
- **Reduces manual QA time**: Instead of manually clicking through your app after every change, you run a test suite in seconds.

Without tests, you're flying blind. With tests, you have a safety net.

## How it works

### unittest workflow

```
1. Import unittest
2. Create a class that inherits unittest.TestCase
3. Write methods starting with "test_" 
4. Use self.assertEqual(), self.assertTrue(), etc. to make assertions
5. Run: python -m unittest test_file.py
```

### pytest workflow

```
1. Create a file starting with "test_" (e.g., test_math.py)
2. Write plain functions starting with "test_"
3. Use plain "assert" statements
4. Run: pytest (discovers and runs all tests automatically)
```

**Test discovery:**
Both frameworks look for files matching `test_*.py` or `*_test.py` and within those, functions/classes starting with `test`.

**The testing cycle (Red-Green-Refactor):**
1. **Red**: Write a test that fails (the feature doesn't exist yet).
2. **Green**: Write the minimum code to make the test pass.
3. **Refactor**: Clean up the code while keeping tests green.

This is called **Test-Driven Development (TDD)**.

## Internal working

**How `pytest` discovers and runs tests:**

1. `pytest` recursively walks directories (starting from current dir) looking for files matching `test_*.py` or `*_test.py`.
2. In each file, it finds functions/classes whose names start with `test`.
3. For each test function, pytest:
   - Sets up fixtures (if any) by calling the fixture functions.
   - Executes the test function (passing fixtures as parameters).
   - Catches any unhandled exceptions (including `AssertionError` from failed asserts).
   - Records pass/fail/skip status with detailed traceback.
4. After all tests, pytest prints a summary: how many passed, failed, skipped.

**How `unittest` works:**
- `unittest.TestCase` uses `setUp()` and `tearDown()` methods that run before/after each test.
- Assertions (`assertEqual`, `assertTrue`, etc.) are methods that raise `AssertionError` on failure.
- A `TestLoader` discovers tests and a `TestRunner` executes them and produces results.

**How mocking works:**
- `unittest.mock.Mock` creates a fake object that records how it's called.
- `patch` temporarily replaces a real object/function with a Mock during a test.
- When the code under test calls the patched object, the Mock records the call and returns a configured value.
- After the test, `patch` restores the original object.

This lets you test code that depends on external services (APIs, databases, filesystems) without actually calling them.

## Real-world analogy

**Testing is like a safety inspector at a car factory:**

- **Without tests**: You build a car, ship it, and hope it doesn't crash. If it does, you recall thousands of cars (expensive!).
- **With tests**: Every component is tested individually:
  - **Unit tests**: Test the brakes alone (a single function).
  - **Integration tests**: Test brakes + wheels together (modules working together).
  - **End-to-end tests**: Drive the car on a test track (full application).
- **Mocking**: Instead of installing real brakes for testing, you use a simulation that measures: "Was the stop signal sent? Was pressure applied?" The simulation (mock) is faster, safer, and doesn't require a full car.

**Red-Green-Refactor** is like cooking with a recipe:
1. **Red**: Read the recipe (write the test for the dish you want).
2. **Green**: Cook the dish (write code until it tastes right).
3. **Refactor**: Clean up the kitchen (improve code quality without changing taste).

## Examples (with code)

### 1. unittest example

```python
import unittest

def add(a, b):
    return a + b

def divide(a, b):
    if b == 0:
        raise ValueError("Cannot divide by zero")
    return a / b

class TestMathOperations(unittest.TestCase):
    
    def setUp(self):
        """Runs before EACH test. Good for creating test data."""
        self.numbers = [1, 2, 3, 4, 5]
    
    def tearDown(self):
        """Runs after EACH test. Good for cleanup."""
        pass
    
    def test_add_positive(self):
        result = add(2, 3)
        self.assertEqual(result, 5)
    
    def test_add_negative(self):
        self.assertEqual(add(-1, 1), 0)
    
    def test_add_zero(self):
        self.assertEqual(add(0, 0), 0)
    
    def test_divide_normal(self):
        self.assertAlmostEqual(divide(10, 3), 3.333, places=3)
    
    def test_divide_by_zero(self):
        with self.assertRaises(ValueError):
            divide(5, 0)
    
    def test_list_length(self):
        self.assertEqual(len(self.numbers), 5)

if __name__ == "__main__":
    unittest.main()
```

Run: `python -m unittest test_math.py`

### 2. pytest example (simpler, more modern)

```python
# test_operations.py
import pytest

def add(a, b):
    return a + b

def divide(a, b):
    if b == 0:
        raise ValueError("Cannot divide by zero")
    return a / b

def test_add_positive():
    assert add(2, 3) == 5

def test_add_negative():
    assert add(-1, 1) == 0

def test_add_strings():
    assert add("a", "b") == "ab"

def test_divide_normal():
    assert divide(10, 3) == pytest.approx(3.333, rel=1e-3)

def test_divide_by_zero():
    with pytest.raises(ValueError, match="Cannot divide by zero"):
        divide(5, 0)

@pytest.mark.parametrize("a,b,expected", [
    (1, 2, 3),
    (0, 0, 0),
    (-1, -1, -2),
    (100, 200, 300),
])
def test_add_parametrized(a, b, expected):
    assert add(a, b) == expected
```

Run: `pytest test_operations.py -v`

### 3. pytest fixtures

```python
# test_database.py
import pytest

class Database:
    def __init__(self):
        self.data = {}
    
    def insert(self, key, value):
        self.data[key] = value
    
    def get(self, key):
        return self.data.get(key)
    
    def clear(self):
        self.data.clear()

@pytest.fixture
def db():
    """Create a fresh database for each test."""
    database = Database()
    database.insert("default", "test")
    yield database  # Test runs here
    database.clear()  # Cleanup after test

def test_insert(db):
    db.insert("name", "Alice")
    assert db.get("name") == "Alice"

def test_get_default(db):
    assert db.get("default") == "test"

def test_get_missing(db):
    assert db.get("nonexistent") is None

# Fixture scope: "function" (default), "class", "module", "session"
@pytest.fixture(scope="session")
def config():
    """Loaded once per test session."""
    return {"api_url": "https://api.example.com", "timeout": 5}
```

### 4. pytest.approx and pytest.raises

```python
import pytest

def test_float_comparison():
    # DON'T: assert 0.1 + 0.2 == 0.3  (floating point)
    # DO:
    assert 0.1 + 0.2 == pytest.approx(0.3)
    assert (0.1 + 0.2) == pytest.approx(0.3, rel=1e-9)  # relative tolerance
    assert 100.0 == pytest.approx(100.5, abs=1.0)  # absolute tolerance

def test_exception():
    with pytest.raises(ValueError) as exc_info:
        int("not_a_number")
    assert "invalid literal" in str(exc_info.value)
```

### 5. Mocking with unittest.mock

```python
# app.py (the code we want to test)
import requests

def fetch_user(user_id):
    response = requests.get(f"https://api.example.com/users/{user_id}")
    response.raise_for_status()
    return response.json()

# test_app.py
from unittest.mock import patch, Mock
import app

@patch("app.requests.get")
def test_fetch_user_success(mock_get):
    # Set up the mock
    mock_response = Mock()
    mock_response.status_code = 200
    mock_response.json.return_value = {"id": 1, "name": "Alice"}
    mock_get.return_value = mock_response
    
    # Call the function (which uses requests.get internally)
    result = app.fetch_user(1)
    
    # Assertions
    assert result["name"] == "Alice"
    mock_get.assert_called_once_with("https://api.example.com/users/1")

@patch("app.requests.get")
def test_fetch_user_http_error(mock_get):
    mock_response = Mock()
    mock_response.raise_for_status.side_effect = requests.exceptions.HTTPError("404")
    mock_get.return_value = mock_response
    
    with pytest.raises(requests.exceptions.HTTPError):
        app.fetch_user(999)
```

### 6. MagicMock — more capable mock

```python
from unittest.mock import MagicMock

# MagicMock supports magic methods like __len__, __iter__, etc.
mock_obj = MagicMock()
mock_obj.__len__.return_value = 5
assert len(mock_obj) == 5

mock_obj.__iter__.return_value = iter([1, 2, 3])
assert list(mock_obj) == [1, 2, 3]

mock_obj.some_method.return_value = 42
assert mock_obj.some_method(1, 2, key="val") == 42
mock_obj.some_method.assert_called_once_with(1, 2, key="val")
```

### Creative mini project: Test-driven calculator development

Let's build a calculator using **Red-Green-Refactor** TDD.

**Step 1 (RED): Write tests first**

```python
# test_calculator.py
import pytest
from calculator import Calculator

@pytest.fixture
def calc():
    return Calculator()

def test_add(calc):
    assert calc.add(2, 3) == 5
    assert calc.add(-1, 1) == 0
    assert calc.add(0, 0) == 0

def test_subtract(calc):
    assert calc.subtract(5, 3) == 2
    assert calc.subtract(0, 5) == -5
    assert calc.subtract(-3, -3) == 0

def test_multiply(calc):
    assert calc.multiply(4, 3) == 12
    assert calc.multiply(0, 100) == 0
    assert calc.multiply(-2, 3) == -6

def test_divide(calc):
    assert calc.divide(10, 2) == 5
    assert calc.divide(7, 3) == pytest.approx(2.333, rel=1e-3)

def test_divide_by_zero(calc):
    with pytest.raises(ValueError, match="Cannot divide by zero"):
        calc.divide(5, 0)

def test_power(calc):
    assert calc.power(2, 3) == 8
    assert calc.power(5, 0) == 1
    assert calc.power(2, -1) == 0.5

def test_factorial(calc):
    assert calc.factorial(0) == 1
    assert calc.factorial(1) == 1
    assert calc.factorial(5) == 120

def test_factorial_negative(calc):
    with pytest.raises(ValueError, match="Cannot factorial"):
        calc.factorial(-3)

@pytest.mark.parametrize("a,b,expected", [
    (2, 3, 5),
    (100, 200, 300),
    (-5, 5, 0),
])
def test_add_parametrized(calc, a, b, expected):
    assert calc.add(a, b) == expected
```

**Step 2 (GREEN): Write code to pass tests**

```python
# calculator.py
class Calculator:
    def add(self, a, b):
        return a + b
    
    def subtract(self, a, b):
        return a - b
    
    def multiply(self, a, b):
        return a * b
    
    def divide(self, a, b):
        if b == 0:
            raise ValueError("Cannot divide by zero")
        return a / b
    
    def power(self, a, b):
        return a ** b
    
    def factorial(self, n):
        if n < 0:
            raise ValueError("Cannot factorial negative numbers")
        if n <= 1:
            return 1
        return n * self.factorial(n - 1)
```

**Step 3 (REFACTOR): Improve without breaking tests**

```python
# calculator.py (refactored)
from functools import reduce
import operator

class Calculator:
    def add(self, *args):
        return sum(args)
    
    def subtract(self, a, b):
        return a - b
    
    def multiply(self, *args):
        return reduce(operator.mul, args, 1)
    
    def divide(self, a, b):
        if b == 0:
            raise ValueError("Cannot divide by zero")
        return a / b
    
    def power(self, a, b):
        return a ** b
    
    def factorial(self, n):
        if n < 0:
            raise ValueError("Cannot factorial negative numbers")
        if n <= 1:
            return 1
        return n * self.factorial(n - 1)
```

Run: `pytest test_calculator.py -v`

## Common mistakes

- **Not running tests**: Tests only work if you run them. Set up CI to run tests on every push.
- **Testing implementation, not behaviour**: Don't test *how* something works internally; test *what* it produces. Refactoring shouldn't break tests.
- **Testing everything except the tricky parts**: Write tests for edge cases (empty inputs, zeros, negatives, special characters, boundary values).
- **Brittle tests**: Tests that depend on exact string formatting, network access, or specific dates fail for the wrong reasons. Use mocks and fixtures.
- **No isolation**: Tests that share state (e.g., modify global variables, files, database) can fail when run in different orders. Each test should be independent.
- **Too many assertions per test**: A test should ideally test one thing. If it has 10 assertions and the first fails, you don't know about the other 9.
- **Testing external APIs directly**: Use mocking. Your tests should not depend on network availability or API rate limits.
- **Forgetting `__init__.py` in test directories**: pytest usually handles this, but if you see import errors, add empty `__init__.py` files.
- **Using `assertTrue` instead of `assertEqual`**: When a test fails, `assertEqual` shows you both the expected and actual values. `assertTrue` just says "it was False".

## Interview questions + answers

**Q1: What's the difference between `unittest` and `pytest`?**
A: `unittest` is built-in, class-based, uses assertion methods (`self.assertEqual()`). `pytest` is third-party, function-based, uses plain `assert`, has fixtures, parametrization, and auto-discovery. `pytest` is more concise and powerful.

**Q2: Explain mocking and why it's useful.**
A: Mocking replaces real objects with fake ones that record calls and return configured values. It's useful for testing code that depends on external services (APIs, databases) without actually calling them — making tests faster, deterministic, and offline-capable.

**Q3: What is the Red-Green-Refactor cycle?**
A: Red = write a failing test. Green = write minimum code to pass. Refactor = improve code quality while keeping tests green. This is the core of Test-Driven Development (TDD).

**Q4: How do you test a function that raises an exception in pytest?**
A: `with pytest.raises(ValueError, match="error message"): function_that_raises()`

**Q5: What is a pytest fixture?**
A: A fixture is a function decorated with `@pytest.fixture` that provides test data or setup. It runs before each test (by default) and can do cleanup with `yield`. Fixtures promote reuse and reduce duplication.

**Q6: How would you test floating-point calculations?**
A: Never use `assert result == 0.3` (floating point precision). Use `pytest.approx()`: `assert result == pytest.approx(0.3, rel=1e-9)`.

**Q7: What is test coverage and how do you measure it?**
A: Test coverage measures what percentage of your code lines are executed during tests. Use `pytest --cov=src` (with `pytest-cov` installed) to see a coverage report.

**Q8: When should you use `side_effect` vs `return_value` on a Mock?**
A: `return_value` sets what the mock returns when called. `side_effect` can be: (1) an exception to raise, (2) a list of return values (one per call), (3) a function that computes return values dynamically.

**Q9: What is parametrization in pytest?**
A: `@pytest.mark.parametrize("arg1,arg2", [(val1, val2), ...])` runs the test function multiple times with different arguments. It avoids duplicating test code.

**Q10: What's the difference between `assert_called_once` and `assert_called_once_with`?**
A: `assert_called_once()` checks the mock was called exactly once (any args). `assert_called_once_with(1, 2)` checks it was called exactly once with those specific arguments.

## Knowledge check (5-10 questions with answers)

**Q1: Write a pytest test that checks `add(2, 3)` returns `5`.**
A: 
```python
def test_add():
    assert add(2, 3) == 5
```

**Q2: What naming convention must pytest test files and functions follow?**
A: Files: `test_*.py` or `*_test.py`. Functions: start with `test_`.

**Q3: How do you run all tests in the current directory with pytest?**
A: Run `pytest` (with no arguments) in the directory.

**Q4: What does `@patch("module.function_name")` do?**
A: It temporarily replaces `module.function_name` with a `MagicMock` object during the test. The original is restored after the test.

**Q5: How do you test that a function raises a `ValueError`?**
A: `with pytest.raises(ValueError): function_that_raises()`

**Q6: What is the purpose of `setUp()` in `unittest.TestCase`?**
A: It runs before each test method to set up test data or state.

**Q7: What does `pytest --cov=myapp --cov-report=html` do?**
A: Measures test coverage of the `myapp` package and generates an HTML report.

**Q8: Write a parametrized pytest test that tests `double(2)==4`, `double(0)==0`, `double(-3)==-6`.**
A:
```python
@pytest.mark.parametrize("input,expected", [(2, 4), (0, 0), (-3, -6)])
def test_double(input, expected):
    assert double(input) == expected
```

**Q9: What's the difference between `return_value` and `side_effect` on a Mock?**
A: `return_value` returns a value when the mock is called. `side_effect` can raise exceptions, return different values per call, or call a function.

**Q10: True or False: Tests should depend on each other (one test sets up data for another).**
A: False. Each test must be independent and self-contained. Test order should not matter.

## Revision notes (bullet points summary)

- Testing automates verification of your code's correctness.
- `unittest` (built-in, class-based) vs `pytest` (third-party, function-based, simpler).
- pytest uses plain `assert`; files/functions must start with `test_`.
- Fixtures (`@pytest.fixture`) provide reusable test setup/teardown.
- Parametrization (`@pytest.mark.parametrize`) runs a test with multiple inputs.
- Mocking (`@patch`, `Mock`, `MagicMock`) replaces external dependencies.
- `approx()` for float comparisons, `pytest.raises()` for expected exceptions.
- Red-Green-Refactor: write failing test → make it pass → improve code.
- Coverage (`pytest-cov`) shows untested code lines.
- Each test should be independent and test one behaviour.
