# Environment Variables

## Definition

**Environment variables** are key-value pairs stored outside your code, in the operating system or a local file (`.env`), that configure how your application runs. Think of them as a global settings dictionary your program can read — but not hardcode.

Common examples: `DATABASE_URL`, `API_KEY`, `SECRET_KEY`, `DEBUG=True`, `EMAIL_HOST`.

In Python, you access them via `os.environ` (a dict-like object) or `os.getenv()`. The `python-dotenv` library lets you load variables from a `.env` file for local development.

## Why it exists

- **Security**: API keys, passwords, database URLs, and tokens must NEVER be committed to version control. Environment variables keep secrets out of your codebase.
- **Portability**: The same code runs in development, staging, and production — just with different environment variables. No code changes needed between environments.
- **Configuration without code changes**: Change behavior (debug mode, log level, database) by flipping an environment variable — no deployment needed.
- **12-Factor App standard**: The industry best practice for modern applications is to store config in environment variables.

## How it works

1. Before running your Python program, you set environment variables in your shell or in a `.env` file.
2. Your Python code reads them with `os.getenv("KEY")` or `os.environ["KEY"]`.
3. If running locally with a `.env` file, `python-dotenv` reads the file and populates `os.environ` when you call `load_dotenv()`.
4. In production, your hosting platform (Heroku, AWS, Docker, etc.) sets real environment variables directly — you never need a `.env` file there.

```
Development:  .env file  --> load_dotenv() --> os.environ
Production:   Platform   --> (set natively) --> os.environ
                             Your code reads from os.environ
```

## Internal working

**`os.environ`** is a Python `os._Environ` object — a mapping that wraps the C-level `environ` variable from the POSIX standard (or Windows equivalent). When you do `os.environ["HOME"]`, Python calls `getenv("HOME")` from the C runtime library, which looks up the value from the process's environment block.

- Environment variables are **inherited by child processes**. When your shell runs `python myscript.py`, the Python process inherits all the shell's environment variables.
- Setting `os.environ["MY_VAR"] = "value"` modifies the current process's environment (and any child processes it spawns), but does NOT persist in the shell or system after the process exits.
- `os.getenv("KEY", "default")` is safer than `os.environ["KEY"]` because it doesn't raise `KeyError` if the variable is missing.
- `python-dotenv` reads `.env` line by line, parses `KEY=VALUE` pairs, and calls `os.environ.setdefault(KEY, VALUE)` or direct assignment for each. It does NOT overwrite already-set environment variables (unless you pass `override=True`).

**`.env` file format:**
```
# Comment (lines starting with #)
DATABASE_URL=postgresql://user:pass@localhost:5432/mydb
API_KEY=abc123def456
DEBUG=true
MAX_CONNECTIONS=100
```

Important syntax rules:
- No spaces around `=` (usually): `KEY=value` not `KEY = value`
- Values with spaces or special chars: `SECRET="my long pass"`
- Multiline values: `PRIVATE_KEY="-----BEGIN...\n...-----"`
- Variable expansion: `URL=${BASE_URL}/api/v1` (if the parser supports it — most do)

## Real-world analogy

Think of environment variables like **a locker room key tag**:

- You don't keep your home address written on your forehead (hardcoding secrets).
- Before you leave (run your app), you grab the tag (env var) that tells you which locker is yours.
- The tag doesn't contain the address — it just references where the actual information is stored.
- Different lockers (development, staging, production) contain different items, but the tag system works the same way.
- You never write down your locker combination in your training manual (commit secrets).

Or, even simpler: environment variables are like the **control panel on a spaceship** — the same ship flies differently depending on how you set the dials:

| Dial          | Setting      | Behaviour                    |
|---------------|--------------|------------------------------|
| DEBUG         | True         | Shows detailed error logs    |
| DEBUG         | False        | Shows user-friendly errors   |
| DATABASE_URL  | localhost    | Connects to local DB         |
| DATABASE_URL  | prod-server  | Connects to production DB    |

Same code, different environment variables → different behaviour.

## Examples (with code)

### 1. Basic: Reading environment variables

```python
import os

# Safe access with default
home = os.getenv("HOME", "/tmp")
print(f"Home directory: {home}")

# Unsafe (raises KeyError if missing)
# db_url = os.environ["DATABASE_URL"]

# Safe with check
api_key = os.getenv("API_KEY")
if not api_key:
    print("WARNING: API_KEY not set!")
    api_key = "default-dev-key"
```

### 2. Setting environment variables at runtime

```python
import os

# Set for this process (and child processes)
os.environ["MY_APP_MODE"] = "testing"
os.environ["DATABASE_URL"] = "sqlite:///test.db"

# Can also set via direct assignment
os.environ["MAX_RETRIES"] = "3"  # NOTE: Values are always strings!

print(os.getenv("MY_APP_MODE"))  # "testing"
```

### 3. Using python-dotenv

First install: `pip install python-dotenv`

```python
from dotenv import load_dotenv
import os

# Load .env file from current directory
load_dotenv()

# Or specify path: load_dotenv("/path/to/.env")

db_url = os.getenv("DATABASE_URL")
api_key = os.getenv("API_KEY")
debug = os.getenv("DEBUG", "False").lower() == "true"

print(f"DB: {db_url}")
print(f"Debug mode: {debug}")
```

**.env file:**

### 4. Typed access (env vars are always strings)

```python
import os

# All env vars are strings — you must convert
port = int(os.getenv("PORT", "8000"))
debug = os.getenv("DEBUG", "False").lower() in ("true", "1", "yes")
max_connections = int(os.getenv("MAX_CONNECTIONS", "10"))
```

### 5. Type-safe helper function

```python
import os

def get_env_bool(key: str, default: bool = False) -> bool:
    """Read an env var and parse it as boolean."""
    val = os.getenv(key)
    if val is None:
        return default
    return val.lower() in ("true", "1", "yes", "on")

def get_env_int(key: str, default: int = 0) -> int:
    """Read an env var and parse as int."""
    val = os.getenv(key)
    if val is None:
        return default
    try:
        return int(val)
    except ValueError:
        return default
```

### 6. Configuration class using env vars

```python
import os
from dataclasses import dataclass

@dataclass
class Config:
    debug: bool
    database_url: str
    api_key: str
    port: int
    log_level: str

    @classmethod
    def from_env(cls):
        return cls(
            debug=os.getenv("DEBUG", "False").lower() == "true",
            database_url=os.getenv("DATABASE_URL", "sqlite:///app.db"),
            api_key=os.getenv("API_KEY", ""),
            port=int(os.getenv("PORT", "8000")),
            log_level=os.getenv("LOG_LEVEL", "INFO"),
        )

# Usage
config = Config.from_env()
print(config)
```



### Creative mini project: Configurable weather app using .env

```python
# weather_app.py
import os
import requests
from dotenv import load_dotenv

load_dotenv()

API_KEY = os.getenv("WEATHER_API_KEY")
CITY = os.getenv("WEATHER_CITY", "London")
UNITS = os.getenv("WEATHER_UNITS", "metric")  # metric or imperial

def get_weather():
    if not API_KEY:
        print("ERROR: WEATHER_API_KEY not set. Create a .env file.")
        return

    url = "https://api.openweathermap.org/data/2.5/weather"
    params = {
        "q": CITY,
        "appid": API_KEY,
        "units": UNITS,
    }

    try:
        resp = requests.get(url, params=params, timeout=10)
        resp.raise_for_status()
        data = resp.json()

        temp = data["main"]["temp"]
        condition = data["weather"][0]["description"]
        humidity = data["main"]["humidity"]
        unit_symbol = "°C" if UNITS == "metric" else "°F"

        print(f"Weather in {CITY}:")
        print(f"  Temperature: {temp}{unit_symbol}")
        print(f"  Condition: {condition}")
        print(f"  Humidity: {humidity}%")

    except requests.exceptions.RequestException as e:
        print(f"Error fetching weather: {e}")

if __name__ == "__main__":
    get_weather()
```

**.env:**
```
WEATHER_API_KEY=your_actual_api_key_here
WEATHER_CITY=New York
WEATHER_UNITS=imperial
```

The app reads all its config from `.env`. Change the city or units without touching a line of code!

## Common mistakes

- **Committing `.env` to git**: Add `.env` to `.gitignore` immediately. If you already committed it, use `git rm --cached .env` and rotate all exposed secrets.
- **Spaces around `=`**: `KEY = value` doesn't work in most `.env` parsers (the key becomes `"KEY "` with a space). Write `KEY=value`.
- **Quoting values incorrectly**: `PASSWORD=my$ecret` — the `$` might be interpreted as variable expansion. Use quotes: `PASSWORD="my$ecret"`.
- **Forgetting `load_dotenv()`**: The `.env` file is NOT automatically loaded. You must call `load_dotenv()` in your code.
- **Not providing defaults**: `os.environ["API_KEY"]` crashes if the var is missing. Always use `os.getenv("API_KEY", "fallback")`.
- **Assuming types**: `os.getenv("PORT")` returns the string `"8000"`, not the integer `8000`. Convert explicitly.
- **Mixing up environment files**: Have separate `.env.development`, `.env.staging`, `.env.production` and load the right one.
- **Hardcoding fallback values in production**: Your production code should crash if critical env vars are missing — fallbacks can mask misconfiguration.

## Interview questions + answers

**Q1: What's the difference between `os.environ["KEY"]` and `os.getenv("KEY")`?**
A: `os.environ["KEY"]` raises `KeyError` if the variable is missing. `os.getenv("KEY", default)` returns `None` (or your default) without raising an error. `os.getenv` is the safer choice.

**Q2: Can you set an environment variable in Python and have it persist after the script ends?**
A: No. Environment variables set with `os.environ["KEY"] = "val"` only affect the current process and its children. Once the script exits, the change is lost. To persist, modify shell config files (`.bashrc`, `.zshrc`) or use a `.env` file.

**Q3: Why should you NEVER hardcode secrets in source code?**
A: (1) Secrets in code get committed to git and exposed to anyone with repo access. (2) Different environments need different secrets; hardcoding prevents this. (3) Rotating a secret requires a code change and redeploy instead of just updating an env var.

**Q4: What is the 12-Factor App principle regarding config?**
A: The 12-Factor App manifesto states: "Store config in the environment." Code should be the same across all environments; only the environment variables change. Never store config as constants in the code.

**Q5: How does `python-dotenv` decide whether to overwrite existing environment variables?**
A: By default, `load_dotenv()` does NOT overwrite variables already set in the environment (because `os.environ.setdefault` is used). Pass `override=True` to force overwrite.

**Q6: What's a good naming convention for environment variables?**
A: UPPER_SNAKE_CASE with a prefix if the app has many env vars (e.g. `MYAPP_DATABASE_URL`, `MYAPP_LOG_LEVEL`). Be descriptive: `SMTP_PASSWORD` not just `PASS`.

**Q7: How would you handle default values for a boolean env var?**
A: `os.getenv("DEBUG", "False").lower() == "true"`. This ensures anything other than "true" (including no value) is treated as False.

**Q8: What is the `override` parameter in `load_dotenv()`?**
A: If `override=True`, values from the `.env` file will overwrite already-set environment variables. Default is `False` (existing env vars take precedence).

## Knowledge check (5-10 questions with answers)

**Q1: Write the code to safely read the environment variable `DATABASE_URL` with a fallback of `sqlite:///default.db`.**
A: `os.getenv("DATABASE_URL", "sqlite:///default.db")`

**Q2: What must you do before calling `os.getenv("API_KEY")` when using a `.env` file?**
A: Call `load_dotenv()` from the `dotenv` module to load the file into `os.environ`.

**Q3: What is the correct `.env` file syntax? (give an example)**
A: `DATABASE_URL=postgresql://localhost/db` (one key=value per line, no spaces around `=`, comments start with `#`).

**Q4: What entry should you add to `.gitignore` for environment variables?**
A: `.env` (and `.env.*` for environment-specific files).

**Q5: What type does `os.getenv("PORT")` return?**
A: A string. You must convert to `int` explicitly if you need a number.

**Q6: Can `load_dotenv()` find a `.env` file in a parent directory?**
A: No, by default it looks in the current working directory. Pass the full path: `load_dotenv("/path/to/.env")`.

**Q7: What happens to env vars set with `os.environ` when the Python process ends?**
A: They are discarded. Environment variables only exist for the lifetime of the process.

**Q8: Name three pieces of information that should NEVER be hardcoded.**
A: API keys, database passwords, secret keys / JWT secrets / any credentials.

**Q9: How do you set an environment variable for a single command in Linux?**
A: `KEY=VALUE python script.py`

**Q10: What's wrong with `os.getenv("COUNT") + 5`?**
A: `os.getenv` returns a string. You need `int(os.getenv("COUNT", "0")) + 5`.

## Revision notes (bullet points summary)

- Environment variables keep secrets and config out of your source code.
- Access with `os.getenv("KEY", "default")` (safe) or `os.environ["KEY"]` (raises KeyError).
- `os.environ` is modified in-process only — changes don't persist after exit.
- Use `python-dotenv` (`.env` files) for local development: install, create `.env`, call `load_dotenv()`.
- `.env` format: `KEY=VALUE`, no spaces around `=`, `#` for comments.
- ALL values are strings — convert to `int`/`bool`/`float` explicitly.
- NEVER commit `.env` files — add to `.gitignore`.
- Following the 12-Factor App principle makes your code portable across environments.
- Use configuration classes/dataclasses to organize env vars.
- Platform-specific setup: `export` on Linux/macOS, `set` on Windows CMD, `$env:` on PowerShell.
