# 4. Threading

## Definition

**Threading** is a concurrency technique where a single process runs multiple **threads**  -  lightweight sub-processes that share the same memory space. Each thread executes independently but can communicate with others through shared data.

```python
import threading
import time

def worker(name):
    print(f"Thread {name} starting")
    time.sleep(1)
    print(f"Thread {name} finished")

t = threading.Thread(target=worker, args=("A",))
t.start()   # Begin execution
t.join()    # Wait for completion
```

## Why it exists

Threading exists to handle **I/O-bound** tasks concurrently  -  where the program spends most of its time waiting for external operations (network responses, disk reads, user input).

Benefits over single-threaded code:
- **Concurrent I/O**: Download files, query databases, serve web requests simultaneously
- **Shared memory**: All threads share the same data  -  no need for IPC
- **Responsive UIs**: Keep the GUI responsive while doing background work

Threading is NOT for CPU-bound tasks (due to the GIL)  -  use multiprocessing instead.

## How it works

1. Create a `Thread` object with a `target` function.
2. Call `t.start()`  -  this spawns the thread and begins executing the target.
3. Optionally call `t.join()`  -  blocks the calling thread until `t` finishes.
4. Threads run **concurrently** (not necessarily parallel), interleaved by the OS scheduler.

```python
def print_numbers():
    for i in range(5):
        print(i)
        time.sleep(0.1)

def print_letters():
    for c in "ABCDE":
        print(c)
        time.sleep(0.1)

t1 = threading.Thread(target=print_numbers)
t2 = threading.Thread(target=print_letters)

t1.start()
t2.start()

t1.join()
t2.join()
# Output interleaves numbers and letters
```

### Daemon threads

Daemon threads run in the background and are **automatically killed** when the main thread exits. Use `daemon=True`.

```python
def background_monitor():
    while True:
        print("Monitoring...")
        time.sleep(1)

t = threading.Thread(target=background_monitor, daemon=True)
t.start()
# Program exits immediately after main thread finishes
```

## Internal working

### Thread Lifecycle

1. **New**: `Thread` object created with `target`
2. **Runnable**: `t.start()` called; thread is ready to run
3. **Running**: OS scheduler gives it CPU time
4. **Blocked/Waiting**: e.g., `time.sleep()`, `lock.acquire()`, I/O wait
5. **Terminated**: target function returns or exception raised

### GIL (Global Interpreter Lock)

CPython has a **GIL**  -  a mutex that prevents multiple threads from executing Python bytecode simultaneously. This means:

- CPU-bound threads run **effectively single-threaded** (one core at a time)
- I/O-bound threads benefit because they release the GIL while waiting
- The GIL is released during I/O operations (`read()`, `write()`, `send()`, `recv()`) and by C extensions (NumPy, Pillow)

```python
# CPU-bound: No speedup with threads
def cpu_heavy():
    for _ in range(10_000_000):
        _ = 2 ** 10

# Multiple threads will NOT finish faster than one thread!
```

## Real-world analogy

**A restaurant kitchen.**

- The main thread is the head chef giving orders.
- Each thread is a line cook working on a station.
- All cooks share the same kitchen (memory space), ingredients (data), and tools (resources).
- **Synchronization issues**: Two cooks reaching for the same knife at the same time (race condition). Solution: a lock (only one chef uses the knife at a time).
- **GIL analogy**: There's only one stove. Cooks can prep ingredients concurrently, but only one uses the stove at a time.

## Examples (with code)

### Basic threads

```python
import threading
import time

def task(name, delay):
    time.sleep(delay)
    print(f"{name} done")

threads = []
for i, delay in enumerate([3, 1, 2]):
    t = threading.Thread(target=task, args=(f"T{i}", delay))
    threads.append(t)
    t.start()

for t in threads:
    t.join()  # Wait for all

print("All done")
```

### Using `ThreadPoolExecutor`

Better than manually managing threads:

```python
from concurrent.futures import ThreadPoolExecutor
import time

def fetch_url(url):
    time.sleep(1)  # simulate I/O
    return f"Data from {url}"

urls = ["http://a.com", "http://b.com", "http://c.com"]

with ThreadPoolExecutor(max_workers=3) as executor:
    results = list(executor.map(fetch_url, urls))

print(results)
```

Or using `submit()` for more control:

```python
with ThreadPoolExecutor(max_workers=3) as executor:
    futures = [executor.submit(fetch_url, url) for url in urls]
    for future in futures:
        result = future.result()  # Blocks until that task completes
        print(result)
```

### Thread synchronization with Lock

```python
counter = 0
lock = threading.Lock()

def increment():
    global counter
    for _ in range(100000):
        with lock:         # Acquire before modifying shared state
            counter += 1   # Critical section
        # Lock released automatically

threads = [threading.Thread(target=increment) for _ in range(5)]
for t in threads: t.start()
for t in threads: t.join()

print(counter)  # 500000 (correct without race condition)
```

**Without the lock**, `counter` would be less than 500000 due to race conditions (read-modify-write interleaving).

### RLock (Reentrant Lock)

Same thread can acquire multiple times without deadlock:

```python
lock = threading.RLock()

def outer():
    with lock:
        inner()     # Same lock, same thread  -  OK with RLock

def inner():
    with lock:
        print("Inside")

outer()  # Would deadlock with a plain Lock
```

### Semaphore

Limits concurrent access to N threads:

```python
semaphore = threading.Semaphore(3)  # Max 3 at a time

def limited_access(name):
    with semaphore:
        print(f"{name} entering")
        time.sleep(1)
        print(f"{name} leaving")

threads = [threading.Thread(target=limited_access, args=(f"T{i}",)) for i in range(10)]
for t in threads: t.start()
for t in threads: t.join()
```

### Queue for thread-safe communication

```python
import queue
import threading
import time

def producer(q):
    for i in range(5):
        q.put(f"item-{i}")
        print(f"Produced item-{i}")
        time.sleep(0.2)

def consumer(q):
    while True:
        item = q.get()
        if item is None:  # Poison pill to signal termination
            break
        print(f"Consumed {item}")
        q.task_done()

q = queue.Queue()
prod = threading.Thread(target=producer, args=(q,))
cons = threading.Thread(target=consumer, args=(q,))

prod.start()
cons.start()

prod.join()
q.put(None)  # Signal consumer to stop
cons.join()
```

## Common mistakes

| Mistake | Why it's wrong | Fix |
|---|---|---|
| Modifying shared data without a lock | Race condition  -  unpredictable results | Use `threading.Lock` |
| Deadlock from nested locks | Thread A holds Lock1, waits Lock2; Thread B holds Lock2, waits Lock1 | Consistent lock ordering, or use `RLock` |
| Not joining threads | Main thread exits before children finish; daemon threads killed abruptly | Always `join()` non-daemon threads |
| Expecting speedup for CPU-bound tasks | GIL prevents parallel execution | Use `multiprocessing` |
| Starting a thread twice | `RuntimeError: threads can only be started once` | Create a new Thread object |
| Using `threading.local()` without understanding | Each thread sees its own version  -  but shared by mistake | Use carefully for thread-local data |

### Race condition demo

```python
# Without lock:
counter = 0
def bad_increment():
    global counter
    for _ in range(100000):
        # Read counter, add 1, write back  -  NOT atomic!
        counter += 1

# Two threads run this; clock cycles interleave the R-M-W, losing updates
```

### Deadlock demo

```python
lock1 = threading.Lock()
lock2 = threading.Lock()

def task_a():
    with lock1:
        time.sleep(0.1)
        with lock2:
            print("A done")

def task_b():
    with lock2:
        time.sleep(0.1)
        with lock1:
            print("B done")

# This may deadlock!
```

## Interview questions + answers

**Q1: What is the GIL? How does it affect threading?**
A: The Global Interpreter Lock is a mutex in CPython that prevents multiple threads from executing Python bytecode simultaneously. For CPU-bound tasks, threading offers no speedup. For I/O-bound tasks, threads are effective because the GIL is released during I/O waits.

**Q2: What is the difference between a Lock and an RLock?**
A: A `Lock` can be acquired only once, even by the same thread (causing deadlock). An `RLock` (reentrant lock) can be acquired multiple times by the same thread without blocking, as long as releases match acquires.

**Q3: What is a race condition?**
A: A race condition occurs when two or more threads access shared data simultaneously, and the final result depends on the timing of their execution. Example: two threads incrementing a counter without synchronization.

**Q4: What is `ThreadPoolExecutor`?**
A: It's a high-level API in `concurrent.futures` for managing a pool of threads. Submit tasks via `.submit()` or `.map()` and get `Future` objects for results. Use `with` for automatic cleanup.

**Q5: What is a daemon thread?**
A: A daemon thread runs in the background and is killed automatically when the main thread exits. Used for monitoring, garbage collection, background tasks. Created with `daemon=True`.

**Q6: How do you share data between threads safely?**
A: Use `threading.Lock` for shared variables, or `queue.Queue` for producer-consumer patterns. Avoid using global mutable state without synchronization.

**Q7: What is `threading.local()`?**
A: It creates thread-local storage  -  each thread sees its own independent copy of the data. Used for per-thread state like database connections.

**Q8: Can you start the same thread twice?**
A: No. Once a thread has finished, it cannot be restarted. Create a new `Thread` object instead.

## Knowledge check (5-10 questions with answers)

1. **How do you start a thread?**
   → Call `thread.start()`.

2. **What does `thread.join()` do?**
   → Blocks the calling thread until `thread` finishes execution.

3. **What is a critical section?**
   → The part of code that accesses shared resources and must not be executed concurrently by multiple threads.

4. **True or False: Threads in Python can run in parallel on multiple CPU cores.**
   → False (for CPU-bound code). The GIL prevents parallel bytecode execution in CPython.

5. **What happens if you don't join a non-daemon thread?**
   → The main thread will wait for it to finish automatically at exit, but it's good practice to join explicitly.

6. **When should you use threading instead of async?**
   → When you have blocking I/O calls that can't be made async, or when you genuinely need parallel execution for I/O.

7. **What is a Semaphore?**
   → A synchronization primitive that limits access to a resource to at most N threads concurrently.

8. **How does `queue.Queue` help with threading?**
   → It provides a thread-safe FIFO queue with blocking `get()` and `put()`, ideal for producer-consumer patterns.

## Revision notes (bullet points summary)

- `threading.Thread(target=func, args=())` creates a thread
- `.start()` begins execution; `.join()` blocks until done
- Daemon threads (`daemon=True`) die with the main thread
- GIL limits CPU-bound threading to one core
- `Lock` prevents race conditions on shared data
- `RLock` allows reentrant acquisition by same thread
- `Semaphore` limits concurrent access count
- `ThreadPoolExecutor` for managing thread pools easily
- `queue.Queue` for safe thread communication
- Threading is for I/O-bound tasks; use multiprocessing for CPU-bound
