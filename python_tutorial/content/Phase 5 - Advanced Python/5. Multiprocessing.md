# 5. Multiprocessing

## Definition

**Multiprocessing** is a parallelism technique that runs code across **multiple processes**  -  each process has its own memory space, its own Python interpreter, and its own GIL. This allows true parallel execution on multi-core CPUs.

```python
import multiprocessing

def worker(name):
    print(f"Process {name} running on CPU {multiprocessing.current_process().pid}")

p = multiprocessing.Process(target=worker, args=("A",))
p.start()
p.join()
```

## Why it exists

Python's threading is limited by the **GIL**  -  only one thread executes bytecode at a time. For **CPU-bound** tasks (number crunching, image processing, data transformation), threading provides no speedup. Multiprocessing bypasses the GIL entirely because each process has its own interpreter.

Multiprocessing is the solution for:
- CPU-intensive computations (parallel processing on multi-core CPUs)
- Isolating workloads (one crashing process doesn't take down others)
- Leveraging all available CPU cores

## How it works

1. Create a `Process` object pointing to a target function.
2. Call `p.start()`  -  the OS forks or spawns a new Python process.
3. The new process runs the target function independently.
4. Call `p.join()` in the parent to wait for completion.

```python
import multiprocessing
import os

def square(n):
    print(f"PID {os.getpid()}: {n} squared = {n*n}")

processes = []
for i in range(4):
    p = multiprocessing.Process(target=square, args=(i,))
    processes.append(p)
    p.start()

for p in processes:
    p.join()
```

Each process runs on a potentially different CPU core, achieving genuine parallelism.

## Internal working

### Process Creation

Two main **start methods**:

| Method | Description | Platform |
|---|---|---|
| **fork** | Parent process is copied  -  child gets all memory at the moment of fork | Linux/macOS |
| **spawn** | New Python interpreter starts from scratch, imports the module | Windows, default on macOS 3.8+ |

```python
multiprocessing.set_start_method("spawn")  # Or "fork" on Linux
```

**Fork** is faster but dangerous (copies all locks, threads, file descriptors). **Spawn** is safer but slower (must re-import modules).

### Memory Isolation

Each process has its own memory space. Modifying a variable in one process does NOT affect the other:

```python
shared_list = []  # This is COPIED on fork/spawn

def append_item(val):
    shared_list.append(val)
    print(f"Inside: {shared_list}")

p = multiprocessing.Process(target=append_item, args=(42,))
p.start()
p.join()
print(f"Outside: {shared_list}")  # Still [] !
```

To share data, you need `Queue`, `Pipe`, `Value`, `Array`, or `Manager`.

## Real-world analogy

**A restaurant with multiple fully-equipped kitchens.**

- Each kitchen (process) has its own stove, fridge, utensils (memory).
- Chefs in different kitchens can cook simultaneously on different burners (CPU cores).
- They cannot grab ingredients from each other's fridges  -  they must pass ingredients through a hatch (Queue/Pipe) or use a shared pantry (Manager/Shared Memory).
- If one kitchen catches fire, the others keep working (process isolation).

This is like threading (multiple cooks in one kitchen) vs multiprocessing (multiple kitchens).

## Examples (with code)

### Basic Process

```python
import multiprocessing
import time

def count_down(name, count):
    while count > 0:
        print(f"{name}: {count}")
        time.sleep(0.5)
        count -= 1

p1 = multiprocessing.Process(target=count_down, args=("A", 3))
p2 = multiprocessing.Process(target=count_down, args=("B", 3))

p1.start()
p2.start()

p1.join()
p2.join()
print("Both done")
```

### Process Pool with `multiprocessing.Pool`

```python
import multiprocessing
import time

def heavy_compute(n):
    """CPU-bound task"""
    total = 0
    for i in range(n * 1_000_000):
        total += i
    return n, total

with multiprocessing.Pool(processes=4) as pool:
    results = pool.map(heavy_compute, [1, 2, 3, 4])

print(results)
```

### `concurrent.futures.ProcessPoolExecutor`

Higher-level API, similar to ThreadPoolExecutor:

```python
from concurrent.futures import ProcessPoolExecutor

def is_prime(n):
    if n < 2:
        return False
    for i in range(2, int(n ** 0.5) + 1):
        if n % i == 0:
            return False
    return True

numbers = [23, 47, 97, 101, 103, 1000000007]

with ProcessPoolExecutor(max_workers=4) as executor:
    results = list(executor.map(is_prime, numbers))

for num, prime in zip(numbers, results):
    print(f"{num}: {'prime' if prime else 'not prime'}")
```

### Pool: `map`, `apply`, `apply_async`

```python
import multiprocessing
import time

def square(x):
    time.sleep(1)
    return x * x

with multiprocessing.Pool(4) as pool:
    # map: block until all results ready
    results = pool.map(square, range(10))    # [0, 1, 4, 9, ...]

    # apply: run one task, block for result
    result = pool.apply(square, (5,))        # 25

    # apply_async: non-blocking, get result later via callback or .get()
    async_result = pool.apply_async(square, (6,))
    print("Task submitted...")
    value = async_result.get()               # Blocks until ready
    print(f"Got: {value}")

    # map_async: non-blocking map
    async_results = pool.map_async(square, range(10))
    async_results.wait()                     # Wait for all
    print(async_results.get())
```

### Sharing data with Queue

```python
import multiprocessing

def producer(q):
    for i in range(5):
        q.put(f"data-{i}")

def consumer(q):
    while True:
        item = q.get()
        if item is None:
            break
        print(f"Got: {item}")

if __name__ == "__main__":
    q = multiprocessing.Queue()
    prod = multiprocessing.Process(target=producer, args=(q,))
    cons = multiprocessing.Process(target=consumer, args=(q,))

    prod.start()
    cons.start()

    prod.join()
    q.put(None)  # Poison pill
    cons.join()
```

### Sharing data with Value and Array

```python
import multiprocessing

def increment(counter):
    for _ in range(10000):
        with counter.get_lock():  # Need lock for Value
            counter.value += 1

if __name__ == "__main__":
    counter = multiprocessing.Value("i", 0)  # 'i' = signed int
    processes = [multiprocessing.Process(target=increment, args=(counter,))
                 for _ in range(4)]
    for p in processes: p.start()
    for p in processes: p.join()
    print(counter.value)  # 40000
```

### Manager for shared state

Manager allows sharing complex Python objects across processes:

```python
import multiprocessing

def worker(d, key, value):
    d[key] = value

if __name__ == "__main__":
    with multiprocessing.Manager() as manager:
        shared_dict = manager.dict()
        jobs = []
        for i in range(5):
            p = multiprocessing.Process(target=worker, args=(shared_dict, i, i**2))
            jobs.append(p)
            p.start()

        for p in jobs:
            p.join()

        print(dict(shared_dict))  # {0: 0, 1: 1, 2: 4, 3: 9, 4: 16}
```

### Pipe for two-way communication

```python
import multiprocessing

def ping(conn):
    conn.send("ping")
    msg = conn.recv()
    print(f"Received: {msg}")
    conn.close()

def pong(conn):
    msg = conn.recv()
    print(f"Received: {msg}")
    conn.send("pong")
    conn.close()

parent_conn, child_conn = multiprocessing.Pipe()
p1 = multiprocessing.Process(target=ping, args=(parent_conn,))
p2 = multiprocessing.Process(target=pong, args=(child_conn,))

p1.start()
p2.start()
p1.join()
p2.join()
```

## Common mistakes

| Mistake | Why it's wrong | Fix |
|---|---|---|
| Not using `if __name__ == "__main__":` on Windows/spawn | Processes re-import the module, creating infinite recursion | Always guard process creation code |
| Expecting shared memory from global variables | Each process has its own memory space | Use `Queue`, `Value`, `Array`, or `Manager` |
| Using `fork` with threads or locks | Forking a multi-threaded process can leave locks in inconsistent state | Use `spawn` start method |
| Heavy overhead for small tasks | Process creation is expensive (spawns new interpreter) | Use `Pool` for task reuse, not per-task processes |
| Forgetting `.get()` on `apply_async` | Task never finishes or results are lost | Always fetch results |
| Sharing file descriptors/sockets across processes | File descriptors are copied on fork but sharing concurrent access is risky | Use Queue for passing connections |

## Interview questions + answers

**Q1: What is the difference between threading and multiprocessing?**
A: Threads share memory and are limited by the GIL (good for I/O). Processes have separate memory and bypass the GIL (good for CPU). Processes have higher overhead to create.

**Q2: What are the start methods for multiprocessing?**
A: `fork` (copy parent process  -  fast but risky, Unix only), `spawn` (new interpreter  -  safe but slow, Windows default), `forkserver` (fork from a clean server process  -  Unix 3.8+).

**Q3: How do you share data between processes?**
A: `multiprocessing.Queue` (FIFO), `Pipe` (two-way), `Value`/`Array` (shared memory), `Manager` (high-level proxy objects).

**Q4: What is `multiprocessing.Pool`?**
A: A process pool that manages a fixed number of worker processes. Use `.map()`, `.apply()`, `.starmap()` to distribute tasks and collect results.

**Q5: What is the difference between `Pool.map` and `Pool.apply_async`?**
A: `map` blocks until all results are ready (synchronous). `apply_async` returns a `AsyncResult` immediately; call `.get()` to retrieve the result (asynchronous).

**Q6: What is `multiprocessing.Manager`?**
A: A manager creates a server process that holds Python objects (lists, dicts, etc.). Other processes access them via proxies. Simpler than shared memory but slower.

**Q7: Can multiprocessing be used on Windows?**
A: Yes, but Windows uses `spawn` (no `fork`). You must guard process creation with `if __name__ == "__main__":`.

**Q8: What is fork bomb and how to avoid it?**
A: A fork bomb happens when processes recursively fork without limit. Avoid by guarding with `if __name__` and having clear termination conditions.

## Knowledge check (5-10 questions with answers)

1. **How do you create a new process?**
   → `p = multiprocessing.Process(target=func)` then `p.start()`.

2. **True or False: Multiprocessing bypasses the GIL.**
   → True. Each process has its own GIL.

3. **What is the main disadvantage of multiprocessing vs threading?**
   → Higher memory and startup overhead (each process has its own Python interpreter).

4. **What does `pool.map(func, iterable)` do?**
   → Distributes items from iterable across pool workers, collects results in order.

5. **How do you get a result from `pool.apply_async`?**
   → Call `.get()` on the returned `AsyncResult` object.

6. **What is the purpose of `if __name__ == "__main__":` in multiprocessing code?**
   → Prevents infinite process spawning when the module is re-imported by child processes.

7. **What does `multiprocessing.Pipe` return?**
   → Two connection objects, one for each end of the pipe.

8. **True or False: Modifying a global list in one process affects the list in other processes.**
   → False. Each process has its own copy of memory.

## Revision notes (bullet points summary)

- `Process`  -  separate OS process with own memory, own GIL
- `.start()` launches; `.join()` waits
- Start methods: `fork` (fast/Unix), `spawn` (slow/safe/Windows)
- `Pool` for managing worker processes efficiently
- `map`, `apply` (blocking) vs `apply_async`, `map_async` (non-blocking)
- Sharing: `Queue`, `Pipe`, `Value`, `Array`, `Manager`
- Use for CPU-bound tasks (bypasses GIL)
- Higher overhead than threading
- Always guard with `if __name__ == "__main__":`
- `ProcessPoolExecutor` for higher-level API
