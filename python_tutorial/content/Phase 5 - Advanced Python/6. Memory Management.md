# 6. Memory Management

## Definition

**Memory management** in Python refers to how Python allocates, uses, and reclaims memory for objects during program execution. It involves:

- **Reference counting** — tracking how many references point to an object
- **Garbage collection** — reclaiming memory from objects that are no longer reachable
- **Memory allocation** — how objects are created and stored in the heap

```python
import sys
import gc

x = [1, 2, 3]
print(sys.getrefcount(x))  # 2 (one from x, one from getrefcount argument)
del x                      # Decrement reference count
gc.collect()               # Force garbage collection
```

## Why it exists

Automatic memory management frees developers from manual malloc/free (C) or new/delete (C++). This eliminates entire categories of bugs:

- **Memory leaks** — forgetting to free memory (though Python can still leak via circular refs + missing GC)
- **Dangling pointers** — accessing freed memory
- **Double free** — freeing memory twice
- **Buffer overflows** — writing past allocated memory

Python handles memory automatically so you can focus on logic, not memory bookkeeping.

## How it works

### Everything is an object

Every value in Python is an object allocated on the **heap**. Variables are just **names** (references) pointing to objects.

```python
a = 42       # 'a' references an int object
b = a        # 'b' references the SAME int object (not a copy)
a = 43       # 'a' now references a new int object; 'b' still points to 42
```

### Mutable vs Immutable

| Category | Types | Characteristics |
|---|---|---|
| **Immutable** | `int`, `float`, `str`, `tuple`, `frozenset`, `bytes` | Cannot be changed after creation; any "modification" creates a new object |
| **Mutable** | `list`, `dict`, `set`, `bytearray`, custom objects | Can be modified in-place without creating a new object |

```python
# Immutable: modification creates new object
s = "hello"
s2 = s
s += " world"   # s now points to new str; s2 still points to "hello"

# Mutable: modification changes in-place
lst = [1, 2, 3]
lst2 = lst
lst.append(4)   # Both lst and lst2 see [1, 2, 3, 4]
```

## Internal working

### Reference Counting

Every Python object has a header field `ob_refcnt` that counts how many references point to it.

```python
import sys
x = "hello"
print(sys.getrefcount(x))  # Typically > 1 (interned string)
y = x
print(sys.getrefcount(x))  # Increased by 1
del y
print(sys.getrefcount(x))  # Decreased by 1
```

When `ob_refcnt` drops to **zero**, the object is immediately deallocated.

Operations that **increase** refcount:
- Assigning a variable: `y = x`
- Passing as argument: `func(x)`
- Appending to a list: `lst.append(x)`
- Object composition: `dict[key] = x`

Operations that **decrease** refcount:
- `del x`
- Variable goes out of scope
- Reassigning: `x = new_value`
- Removing from container: `lst.remove(x)`
- Container itself is destroyed

### Garbage Collection (GC)

Reference counting alone fails for **circular references**:

```python
class Node:
    def __init__(self):
        self.next = None

a = Node()
b = Node()
a.next = b
b.next = a  # Circular reference!

del a
del b  # Refcounts are now 1 each (a.next → b, b.next → a) — memory leak!
```

Python's **garbage collector** (in the `gc` module) handles this. It's a **generational collector** that:

1. Tracks container objects (list, dict, set, tuple, custom objects) in a linked list
2. Periodically searches for **unreachable** object cycles
3. Reclaims their memory

Three **generations**: new objects go to gen 0. If they survive a collection, they move to gen 1, then gen 2. The GC scans younger generations more frequently.

```python
import gc
print(gc.get_threshold())  # (700, 10, 10) = collect gen 0 after 700 allocs, etc.
```

### Memory Allocation

Python uses a private heap managed by `PyMalloc`:

- **Small objects** (< 512 bytes): allocated from arena pools (fast, reduces fragmentation)
- **Large objects**: allocated via `malloc` directly from the OS

```python
print(id(x))  # Memory address of the object
```

### Small Integer Caching

Python caches integers from **-5 to 256**:

```python
a = 256
b = 256
print(a is b)  # True (same cached object)

c = 257
d = 257
print(c is d)  # False (different objects, usually)
```

### String Interning

Python automatically interns certain strings (short identifiers, literals):

```python
a = "hello_world"
b = "hello_world"
print(a is b)  # May be True (interned)

c = "hello " + "world"
d = "hello world"
print(c is d)  # Depends on implementation
```

## Real-world analogy

**A library (memory) with books (objects).**

- **Reference counting**: Each book has a checkout counter. Each time someone borrows it (references it), the counter goes up. Returning it decrements the counter. When the counter hits zero, the book is removed from the library.
- **Circular references**: Person A has Person B's phone number, Person B has Person A's phone number. Neither can be "checked out" by anyone else, but they keep each other alive. The librarian (GC) occasionally walks around and notices these stale mutual references and reclaims them.
- **Small integer caching**: Popular books are kept at the front desk instead of the shelves — always the same copy for everyone.
- **Mutable vs Immutable**: A whiteboard (mutable) can be erased and rewritten. A printed book (immutable) — if you want a different version, you need a new book.

import sys

x = []

y = x
print(sys.getrefcount(x))  # 3

z = x

### `id()` and `is` operator

```python
a = [1, 2, 3]
b = [1, 2, 3]

print(a == b)    # True (same value)
print(a is b)    # False (different objects)
print(id(a), id(b))  # Different memory addresses

c = a
print(c is a)    # True (same object)
```

### Garbage collection demo

```python
import gc

class Node:
    def __init__(self, name):
        self.name = name
        self.ref = None

    def __del__(self):
        print(f"{self.name} destroyed")

# Force GC off
gc.disable()

a = Node("A")
b = Node("B")
a.ref = b
b.ref = a  # Circular reference

del a
del b
# Nothing printed — objects still alive due to circular ref

print("Collecting...")
gc.collect()  # Now both are destroyed
```

### Memory profiling with tracemalloc

```python
import tracemalloc

tracemalloc.start()

# Allocate some memory
data = [list(range(1000)) for _ in range(100)]

snapshot = tracemalloc.take_snapshot()
stats = snapshot.statistics("lineno")

for stat in stats[:5]:
    print(stat)
```

### Memory profiler decorator

```python
import tracemalloc
import functools

def memory_profiler(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        tracemalloc.start()
        result = func(*args, **kwargs)
        snapshot = tracemalloc.take_snapshot()
        stats = snapshot.statistics("lineno")
        
        total = sum(stat.size for stat in stats)
        print(f"{func.__name__} allocated ~{total / 1024:.1f} KB")
        tracemalloc.stop()
        return result
    return wrapper

@memory_profiler
def create_large_list():
    return [list(range(1000)) for _ in range(500)]

create_large_list()
```

### `del` and `__del__`

```python
class Resource:
    def __init__(self, name):
        self.name = name
        print(f"{self.name} acquired")

    def __del__(self):
        print(f"{self.name} released")

r = Resource("file.txt")
# ... do work ...
del r  # Explicit deletion — __del__ called

# __del__ is NOT a replacement for context managers!
# It's called by GC at an unpredictable time.
```

### Small integer caching demonstration

```python
# Integers from -5 to 256 are cached
for i in range(-5, 257):
    a = i
    b = i
    if a is not b:
        print(f"{i} is not cached!")
        break
else:
    print("All integers -5 to 256 are cached")

# Outside range
a = 300
b = 300
print(a is b)  # Usually False (not guaranteed!)
```

### Memory fragmentation avoidance with `__slots__`

```python
class Point:
    __slots__ = ("x", "y")  # No __dict__ — saves memory

    def __init__(self, x, y):
        self.x = x
        self.y = y

# Without __slots__, each Point has a __dict__ (~120 bytes overhead)
# With __slots__, only space for x and y (~8 bytes each)

p = Point(1, 2)
# p.z = 3  # AttributeError: 'Point' object has no attribute 'z'
```

## Common mistakes

| Mistake | Why it's wrong | Fix |
|---|---|---|
| Using `is` for value comparison | `is` compares identity, not equality; `a is 1000` may be False even if `a == 1000` | Use `==` for value comparison |
| Relying on `__del__` for cleanup | `__del__` may never be called (circular refs, interpreter exit) | Use context managers (`with`) or `try/finally` |
| Holding references unintentionally (closures, cyclical data structures) | Objects can't be GC'd, causing memory leaks | Use `weakref` for caches/callbacks |
| Expecting `del x` to free memory immediately | Only decrements refcount; actual deallocation may be delayed by GC | Let Python manage it; use `gc.collect()` for testing |
| Modifying a list while iterating | Can cause items to be skipped or errors | Iterate over a copy: `for item in lst[:]:` |
| Creating many objects in a tight loop | High allocation rate slows GC and memory allocator | Use `__slots__`, object pools, or array-based structures |
| Not knowing about integer/string interning | Assumes objects are always distinct | Use `is` only for singletons (`None`, `True`, `False`) |

## Interview questions + answers

**Q1: How does Python manage memory?**
A: Python uses a combination of reference counting (immediate deallocation when count hits 0) and a generational garbage collector (for circular references). Small objects use a private heap allocator (PyMalloc) for efficiency.

**Q2: What is the difference between mutable and immutable objects?**
A: Mutable objects (list, dict, set) can be changed in-place. Immutable objects (int, str, tuple) cannot — any "change" creates a new object. This affects behavior when passing to functions (`+=` on a string creates a local new variable, but `+=` on a list modifies the original).

**Q3: How does reference counting work?**
A: Every Python object has a counter (`ob_refcnt`). It increments on new references (assignments, function calls, container additions) and decrements on deletions or scope exit. When it reaches 0, the object is immediately deallocated.

**Q4: What is the garbage collector and when is it needed?**
A: The `gc` module handles circular references that reference counting can't resolve. It's a generational collector that periodically scans objects and reclaims unreachable cycles.

**Q5: What is `sys.getrefcount()`?**
A: It returns the reference count of an object. Note that the count is always at least 1 higher than expected because passing the object to `getrefcount` creates a temporary reference.

**Q6: How do you find memory leaks in Python?**
A: Use `tracemalloc`, `gc.get_objects()`, or external tools like `memory_profiler`. Look for objects that should have been freed but are still referenced.

**Q7: What are `__slots__` and why use them?**
A: `__slots__` restricts a class to only the listed attributes, eliminating the per-instance `__dict__`. This reduces memory usage (by ~50-70 bytes per instance) and speeds up attribute access.

**Q8: What is the `id()` function?**
A: It returns the memory address of an object (the identity). Two objects with the same `id` at the same time are the same object. `a is b` is equivalent to `id(a) == id(b)`.

**Q9: Does Python have stack and heap?**
A: Python allocates all objects on the heap. The stack holds local variable references (pointers to heap objects), function call frames, and control flow data.

## Knowledge check (5-10 questions with answers)

1. **What happens when an object's reference count reaches 0?**
   → Its memory is immediately deallocated (freed).

2. **What is the main limitation of reference counting?**
   → It can't handle circular references (objects referencing each other keep their counts > 0).

3. **What range of integers does Python cache?**
   → -5 to 256.

4. **True or False: `a is b` checks if two variables point to the same object.**
   → True.

5. **What is string interning?**
   → Python reuses the same object for identical string literals (usually identifiers and short strings).

6. **How do you force garbage collection?**
   → `import gc; gc.collect()`.

7. **What is the purpose of `weakref`?**
   → It creates a reference that doesn't increase the reference count, allowing an object to be GC'd while still being queryable.

8. **True or False: `del x` always frees the memory used by `x`.**
   → False. It only decrements the reference count. The object is freed only when the count reaches 0.

9. **What is the difference between `__dict__` and `__slots__`?**
   → `__dict__` is a dictionary for instance attributes (flexible but memory-heavy). `__slots__` is a fixed tuple of allowed attribute names (memory-efficient but rigid).
