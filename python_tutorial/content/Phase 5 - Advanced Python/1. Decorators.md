# 1. Decorators

## Definition

A **decorator** is a function that takes another function (or class) as input, extends or modifies its behavior, and returns a function (or class) as output. It is a form of metaprogramming  -  code that manipulates code.

The `@decorator` syntax is syntactic sugar for:

```python
@decorator
def func():
    pass

# is exactly equivalent to:
def func():
    pass
func = decorator(func)
```

## Why it exists

Decorators exist because they provide a clean, reusable way to add cross-cutting concerns to functions or methods without modifying their source code. They help you follow the **Open/Closed Principle**: code should be open for extension but closed for modification.

Common use cases:
- Logging function calls
- Timing execution
- Enforcing access control / authentication
- Caching / memoization results
- Rate limiting API calls
- Input validation
- Registering functions (plugins, signal handlers)

**Without decorators**, you would need to manually wrap every function  -  repetitive, error-prone, and impossible to maintain at scale.

## How it works

At its core, a decorator is a function that:

1. Accepts a callable (the decorated function) as its argument.
2. Defines a **wrapper** function inside that adds behavior before/after the original.
3. Returns the wrapper (or a modified callable) which replaces the original.

```python
def my_decorator(func):
    def wrapper():
        print("Before the function runs")
        func()                          # Call the original
        print("After the function runs")
    return wrapper

@my_decorator
def say_hello():
    print("Hello!")

say_hello()
# Output:
# Before the function runs
# Hello!
# After the function runs
```

## Internal working

When Python encounters `@my_decorator`:

1. It reads the function `say_hello` into memory as a function object.
2. It calls `my_decorator(say_hello)`  -  passing the function object.
3. The decorator executes, creates a `wrapper` closure that captures `func` from the enclosing scope.
4. The decorator **returns** the `wrapper` function.
5. Python rebinds the name `say_hello` to this new `wrapper` function.

```python
# Internal mental model:
say_hello = my_decorator(say_hello)  # Replaces original
```

Because of **closures**, the wrapper retains access to the original `func` even after `my_decorator` has finished executing.

**Metadata problem**: The wrapper replaces the original function, so the original's `__name__`, `__doc__`, `__module__` are lost. Solution: `functools.wraps` copies them over.

```python
import functools

def my_decorator(func):
    @functools.wraps(func)          # <- preserves func's metadata
    def wrapper(*args, **kwargs):
        ...
        return func(*args, **kwargs)
    return wrapper
```

## Real-world analogy

**A gift wrapping service:**

You bring a gift (the function) to a wrapping counter (the decorator). The wrapper adds a box, ribbon, and card (extra behavior) and hands back the wrapped gift  -  still your gift, but with enhancements. You receive it back as a "wrapped" version of itself. You can stack multiple wrappings (stacking decorators).

## Examples (with code)

### Basic logger decorator

```python
import functools

def logger(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        print(f"[LOG] Calling {func.__name__} with args={args} kwargs={kwargs}")
        result = func(*args, **kwargs)
        print(f"[LOG] {func.__name__} returned {result}")
        return result
    return wrapper

@logger
def add(a, b):
    return a + b

add(3, 5)
# [LOG] Calling add with args=(3, 5) kwargs={}
# [LOG] add returned 8
```

### Timer decorator

```python
import time
import functools

def timer(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        elapsed = time.perf_counter() - start
        print(f"{func.__name__} took {elapsed:.4f} seconds")
        return result
    return wrapper

@timer
def slow_function():
    time.sleep(1)
    return "done"

slow_function()  # slow_function took 1.0002 seconds
```

### Decorator with arguments: `@repeat(n)`

This requires a **decorator factory**  -  a function that returns the actual decorator.

```python
import functools

def repeat(num_times):
    def decorator(func):
        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            for _ in range(num_times):
                result = func(*args, **kwargs)
            return result
        return wrapper
    return decorator

@repeat(num_times=3)
def greet(name):
    print(f"Hello {name}")

greet("Alice")
# Hello Alice
# Hello Alice
# Hello Alice
```

The call chain: `repeat(3)` → returns `decorator` → which is called with `greet` → returns `wrapper`.

### Stacking multiple decorators

```python
@timer
@logger
def compute(x, y):
    return x ** y

# Equivalent to: compute = timer(logger(compute))
# Execution order: timer's wrapper calls logger's wrapper calls original compute
```

Decorators are applied **bottom-up** but execute **top-down** (outer decorator runs first, calls inner, which calls original).

### Class-based decorator using `__call__`

```python
import functools

class CountCalls:
    def __init__(self, func):
        functools.update_wrapper(self, func)
        self.func = func
        self.count = 0

    def __call__(self, *args, **kwargs):
        self.count += 1
        print(f"Call {self.count} of {self.func.__name__}")
        return self.func(*args, **kwargs)

@CountCalls
def say_hi():
    print("Hi!")

say_hi()  # Call 1 of say_hi \n Hi!
say_hi()  # Call 2 of say_hi \n Hi!
```

### Memoization (caching) decorator

```python
import functools

def memoize(func):
    cache = {}
    @functools.wraps(func)
    def wrapper(*args):
        if args in cache:
            print(f"Cache hit: {args}")
            return cache[args]
        result = func(*args)
        cache[args] = result
        return result
    return wrapper

@memoize
def fibonacci(n):
    if n < 2:
        return n
    return fibonacci(n - 1) + fibonacci(n - 2)
```

### Authentication decorator (pattern)

```python
import functools

def require_auth(func):
    @functools.wraps(func)
    def wrapper(user, *args, **kwargs):
        if not user.get("authenticated"):
            raise PermissionError("User not authenticated")
        return func(user, *args, **kwargs)
    return wrapper

@require_auth
def view_dashboard(user):
    return "Dashboard data"

# view_dashboard({"authenticated": False})  # PermissionError
```

## Common mistakes

| Mistake | Why it's wrong | Fix |
|---|---|---|
| Forgetting `@functools.wraps` | Loses `__name__`, `__doc__`; breaks introspection | Always use `@functools.wraps(func)` on wrapper |
| Returning `wrapper()` instead of `wrapper` | Executes the function immediately instead of returning the replacement | Return the function object, not call result |
| Nesting too deep for decorator arguments | Forgetting the triple-nested structure (factory → decorator → wrapper) | Use `functools.partial` or a decorator class if it gets complex |
| Modifying mutable defaults across calls | Wrapper `*args` refers to a mutable; risky if shared | Use `*args, **kwargs` safely |
| Applying decorators to class methods without `self` | Wrapper must accept `self` (or use `*args`) | Always define wrapper as `def wrapper(*args, **kwargs)` |
| Not preserving signature for tools | Some tools (inspect, pydantic) rely on function signature | Use `functools.wraps` + consider `decorator` library for full signature |

## Interview questions + answers

**Q1: What is a decorator in Python?**
A: A decorator is a function that receives a callable, wraps it with additional behavior, and returns a new callable. The `@` syntax is syntactic sugar for `func = decorator(func)`.

**Q2: How do you preserve metadata when writing a decorator?**
A: Use `@functools.wraps(func)` on the inner wrapper function. It copies `__name__`, `__doc__`, `__module__`, and `__dict__` from the original to the wrapper.

**Q3: Write a decorator that retries a function up to 3 times if it raises an exception.**
A:

```python
import functools
import time

def retry(max_attempts=3, delay=1):
    def decorator(func):
        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            for attempt in range(1, max_attempts + 1):
                try:
                    return func(*args, **kwargs)
                except Exception as e:
                    print(f"Attempt {attempt} failed: {e}")
                    if attempt == max_attempts:
                        raise
                    time.sleep(delay)
            return None
        return wrapper
    return decorator
```

**Q4: What is the difference between a decorator and a decorator factory?**
A: A decorator takes a function and returns a function. A decorator factory is a function that takes arguments (e.g., `@repeat(3)`) and returns a decorator. The outer function is the factory.

**Q5: Can a decorator be a class? How?**
A: Yes. A class can act as a decorator if it implements `__init__` (receives the function) and `__call__` (makes the instance callable). Example: `class CountCalls` above.

**Q6: How do you apply multiple decorators? What is the order of execution?**
A: Stack them with `@a` above `@b` above `def f()`. Application order (bottom-up): `f = a(b(f))`. Execution order (top-down): `a`'s wrapper runs first, calls `b`'s wrapper, which calls the original `f`.

**Q7: What happens if a decorator doesn't return a function?**
A: The decorated name gets bound to whatever the decorator returns  -  it could be a string, an integer, or anything. This is valid Python but usually breaks expected behavior.

**Q8: How does `functools.wraps` work internally?**
A: It is a decorator that calls `functools.update_wrapper(wrapper, wrapped)`, which copies `__module__`, `__name__`, `__qualname__`, `__doc__`, and `__dict__`, and updates `wrapper.__wrapped__` to point to the original.

## Knowledge check (5-10 questions with answers)

1. **What does `@decorator` do syntactically?**
   → It applies `decorator` to the following function: `func = decorator(func)`.

2. **Why should you use `@functools.wraps(func)`?**
   → It preserves the original function's metadata (`__name__`, `__doc__`, etc.) on the wrapper.

3. **What does a decorator factory return?**
   → A decorator (a function that takes a function and returns a function).

4. **In `@a @b def f(): pass`, which decorator is applied first?**
   → `@b` is applied first (bottom-up): `f = a(b(f))`.

5. **Can you decorate a class?**
   → Yes. You can decorate a class just like a function. The decorator receives the class and returns a modified class (e.g., adding methods or attributes).

6. **True or False: A decorator must always return a callable.**
   → False. There is no enforcement; but if it doesn't return a callable, the decorated name won't be callable, breaking typical usage.

7. **How do you pass arguments to a decorator?**
   → Use a decorator factory: an outer function that takes the arguments and returns the actual decorator.

8. **What is a practical use of class-based decorators?**
   → When you need to maintain state across calls (like counting calls, caching).

## Revision notes (bullet points summary)

- Decorator = function that takes a callable and returns a callable
- `@decorator` is sugar for `func = decorator(func)`
- Wrapper must accept `*args, **kwargs` to be generic
- Always use `@functools.wraps(func)` to preserve metadata
- Decorator factory: extra outer layer for `@decorator(args)`
- Stacking: `@a @b def f()` → `f = a(b(f))`; execution: a → b → original
- Class-based decorators: `__init__` receives func, `__call__` makes it callable
- Common use cases: logging, timing, auth, memoization, retry
- Decorators can be applied to functions, methods, and classes
