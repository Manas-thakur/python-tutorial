# Type Hints

## Definition
Type hints (also called type annotations) are a way to indicate the expected data types of variables, function parameters, and return values in Python. They were introduced in Python 3.5 via PEP 484. The syntax uses `: type` for variables/parameters and `-> type` for return types.

```python
# Basic syntax
name: str = "Alice"
age: int = 25

def greet(name: str) -> str:
    return f"Hello, {name}"
```

## Why it exists
- **Documentation**: Self-documenting code  -  anyone reading knows what types are expected
- **IDE support**: Autocompletion, type checking, and refactoring tools work better
- **Static analysis**: Tools like `mypy`, `pyright`, and `pylance` catch type-related bugs before runtime
- **Team collaboration**: Makes codebases easier to understand and maintain
- **Catching bugs early**: Many type errors are caught at analysis time instead of runtime
- **Refactoring**: Renaming or changing structures is safer with type coverage

## How it works
- Type hints are **entirely optional**  -  Python remains dynamically typed
- They are **not enforced at runtime**  -  Python ignores them during execution
- They are stored in `__annotations__` attribute of functions/classes
- Tools like `mypy` scan your code separately to verify type correctness
- The `typing` module provides advanced types: `Optional`, `Union`, `List`, `Dict`, etc.

```python
# Annotations are stored here
def add(a: int, b: int) -> int:
    return a + b

print(add.__annotations__)  # {'a': int, 'b': int, 'return': int}
```

## Internal working
- Python collects all annotations into a dictionary stored in `__annotations__`
- At runtime, annotations are expressions that get evaluated (unless `from __future__ import annotations` is used, which makes them strings)
- The interpreter does NOT check types  -  `add("hello", 5)` runs without error
- Third-party tools like `mypy` parse the AST (Abstract Syntax Tree) and perform static analysis
- With `from __future__ import annotations` (Python 3.7+), all annotations become strings (postponed evaluation), which solves forward-reference issues and improves performance

```python
# Without future import  -  annotations are evaluated
def func(x: list[int]) -> None:
    pass
# func.__annotations__ = {'x': list[int], 'return': None}

from __future__ import annotations

def func2(x: list[int]) -> None:
    pass
# func2.__annotations__ = {'x': 'list[int]', 'return': 'None'}  # strings!
```

## Real-world analogy
Think of type hints like labels on boxes when moving houses. You write "Kitchen  -  fragile glasses" on a box. The movers **can** ignore the label and throw the box around, but with a label they're more likely to handle it correctly, and anyone else who sees the box knows what's inside. Similarly, type hints label your data so tools and humans handle it correctly  -  but Python itself won't enforce it.

## Examples (with code)

### Basic type hints
```python
# Variable annotations
count: int = 0
name: str = "Python"
price: float = 19.99
is_active: bool = True

# Function with parameters and return type
def multiply(x: int, y: int) -> int:
    return x * y

# Function that returns nothing
def log(message: str) -> None:
    print(f"[LOG]: {message}")
```

### Built-in collection types (Python 3.9+)
```python
# Lists, dicts, tuples, sets
names: list[str] = ["Alice", "Bob", "Charlie"]
scores: dict[str, int] = {"Alice": 95, "Bob": 87}
point: tuple[int, int] = (10, 20)
mixed: tuple[int, str, float] = (1, "hello", 3.14)
unique_ids: set[int] = {101, 202, 303}
```

### Using the typing module (older style, still widely used)
```python
from typing import List, Dict, Tuple, Set, Optional, Union, Any

# Older style (Python 3.5-3.8)
names: List[str] = ["Alice", "Bob"]
scores: Dict[str, int] = {"Alice": 95}

# Optional  -  value can be T or None
def find_user(user_id: int) -> Optional[str]:
    database = {1: "Alice", 2: "Bob"}
    return database.get(user_id)  # Returns str or None

# Union  -  value can be one of multiple types
def process_id(identifier: Union[int, str]) -> str:
    return f"ID: {identifier}"

# Any  -  completely flexible (opt out of type checking)
def debug(value: Any) -> None:
    print(value)
```

### Python 3.10+ syntax
```python
# Union with | operator
def process_id(identifier: int | str) -> str:
    return f"ID: {identifier}"

# Optional with | None
def find_user(user_id: int) -> str | None:
    database = {1: "Alice", 2: "Bob"}
    return database.get(user_id)
```

### Type aliases
```python
# Create readable aliases for complex types
Vector = list[float]
Matrix = list[list[float]]

def add_vectors(a: Vector, b: Vector) -> Vector:
    return [x + y for x, y in zip(a, b)]
```

## Common mistakes

| Mistake | Wrong | Correct |
|---------|-------|---------|
| Confusing `Optional` with default | `def f(x: Optional[int] = None)` is fine | `Optional` doesn't set default  -  it means `None` is allowed |
| Using `List` instead of `list` | `List[int]` (still works) | `list[int]` (preferred in 3.9+) |
| Thinking types are enforced | `add("a", 5)` runs fine | Always use `mypy` separately |
| Forgetting `-> None` | `def f()` for void | `def f() -> None` |
| Annotating but never checking | Writing hints but never running mypy | Run `mypy` in CI or pre-commit |
| Over-annotating | `x: int = 5` (redundant) | Omit when type is obvious from literal |
| Forward references without quotes or future import | `class A: def f(self) -> B: ...` (B undefined) | Use `"B"` or `from __future__ import annotations` |

## Interview questions + answers

**Q1: Are type hints enforced at runtime?**
A: No. Python ignores type hints during execution. They are used by static type checkers (like `mypy`), IDEs, and for documentation.

**Q2: What's the difference between `Optional[str]` and `Union[str, None]`?**
A: They are exactly the same. `Optional[X]` is shorthand for `Union[X, None]`.

**Q3: What does `from __future__ import annotations` do?**
A: It makes all annotations strings (postponed evaluation) instead of evaluated expressions. This solves forward reference issues and improves startup time.

**Q4: How do you annotate a function that takes a callback?**
A: Use `Callable` from `typing`: `from typing import Callable` then `def map(func: Callable[[int], str], items: list[int]) -> list[str]:`

**Q5: What's the difference between `list[int]` and `List[int]`?**
A: `list[int]` (Python 3.9+) uses built-in generics. `List[int]` (from `typing`) is the older style. Both work, but built-in generics are preferred in modern code.

**Q6: Can you use type hints with classes you define?**
A: Yes. You can use your own classes as type annotations: `def process(user: User) -> None:`

**Q7: What is `Any` and when should it be used?**
A: `Any` disables type checking for that value. Use it when the type truly can be anything (e.g., deserialized JSON data, dynamic input).

**Q8: How do you annotate a variable that could be `int` or `float`?**
A: `value: int | float` (Python 3.10+) or `value: Union[int, float]` (older).

## Knowledge check

**Q1:** What does the following code print?
```python
def add(a: int, b: int) -> int:
    return a + b

print(add("hello", " world"))
```
A: `hello world`  -  type hints are not enforced at runtime, concatenation works.

**Q2:** How do you annotate a function parameter that could be a string or None?
A: `def func(x: str | None):` or `def func(x: Optional[str]):`

**Q3:** What attribute stores all annotations of a function?
A: `func.__annotations__`

**Q4:** True or False: Type hints make Python a statically typed language.
A: False. Python remains dynamically typed; type hints are for tooling only.

**Q5:** What is the return type annotation for a function that never returns?
A: `NoReturn` from `typing` (e.g., a function that always raises an exception).

**Q6:** What does `list[int]` mean?
A: A list where every element is of type `int`.

**Q7:** What is a type alias? Give an example.
A: A shorthand for a complex type: `Point = tuple[int, int]`

**Q8:** How do you annotate a dict mapping string keys to integer values?
A: `dict[str, int]`

**Q9:** What's wrong with `def func(x: list[int] = [])`?
A: It's a mutable default argument  -  but that's a runtime issue, not a type hint issue. The type hint itself is fine.

**Q10:** Which tool is most commonly used for static type checking in Python?
A: `mypy`

## Revision notes

- Type hints use `: type` for parameters/variables and `-> type` for return values
- They are **not enforced at runtime**  -  Python ignores them
- Benefits: IDE autocompletion, self-documenting code, static analysis with `mypy`
- `Optional[X]` is shorthand for `Union[X, None]` (Python 3.10+: `X | None`)
- `Union[X, Y]` means X or Y (Python 3.10+: `X | Y`)
- Built-in generics (`list[str]`) work from Python 3.9+; use `typing.List[str]` for older versions
- `from __future__ import annotations` postpones annotation evaluation (makes them strings)
- `Any` disables type checking for a value
- Type hints are stored in `__annotations__` dictionary
- Always run a type checker separately  -  Python doesn't enforce anything
