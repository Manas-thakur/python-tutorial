# 1. Error Handling

## Definition
Error handling in Python is the process of anticipating, detecting, and responding to runtime errors (exceptions) using `try`, `except`, `else`, `finally`, and `raise` blocks. It prevents programs from crashing unexpectedly and allows graceful recovery from errors.

## Why it exists
Programs encounter unexpected situations: missing files, invalid user input, network failures, etc. Without error handling, these cause abrupt crashes with unhelpful tracebacks. Error handling allows you to:
- Keep the program running despite errors
- Show user-friendly messages instead of technical tracebacks
- Clean up resources (close files, release connections) even when errors occur
- Separate normal logic from error-handling logic
- Create meaningful, domain-specific errors for your application

## How it works
Python uses **exceptions** — objects that represent errors. When an error occurs, Python *raises* an exception. If you don't handle it, the program stops and prints a traceback. If you `try` a block of code and `except` the exception, the program continues.

```python
try:
    risky_operation()
except SomeError:
    recovery_code()
else:
    runs_if_no_error()
finally:
    always_runs()
```

The `try` block contains code that might raise an exception. The `except` block catches and handles specific exceptions. The `else` block runs only if no exception occurred. The `finally` block always runs regardless of exceptions.

## Internal working
When Python encounters a `try` statement, it sets up an **exception handler** on the call stack. At runtime:

1. Python executes the `try` block normally.
2. If no exception occurs, it skips all `except` blocks and runs `else` (if present), then `finally`.
3. If an exception occurs, Python stops executing the `try` block immediately and checks each `except` clause in order.
4. The first `except` clause whose exception class matches the raised exception (using `isinstance()`) runs.
5. If no `except` matches, the exception propagates up the call stack to enclosing `try` blocks or eventually to the top-level where Python prints the traceback and exits.
6. The `finally` block executes **before** the exception propagates further (or after `except`/`else` completes).

The `raise` statement creates a new exception instance or re-raises the current one. Internally, it constructs the exception object and begins stack unwinding.

```python
# Internal flow visualization
try:                          # Step 1: Set up exception handler
    x = 1 / 0                 # Step 2: ZeroDivisionError raised
except ZeroDivisionError:     # Step 3: Match found
    print("Cannot divide")    # Step 4: Handler runs
finally:                      # Step 5: Always executes
    print("Cleanup")          # Step 6: Cleanup done
```

## Real-world analogy
A restaurant kitchen:
- **try**: The chef attempts to cook a dish
- **except**: If an ingredient is missing (error), the chef has a backup plan (substitute ingredient, or tell the waiter to inform the customer)
- **else**: If everything goes well, the chef plates the dish beautifully
- **finally**: Regardless of success or failure, the chef cleans the workstation afterward
- **raise**: The chef explicitly sends back a spoiled ingredient to the supplier

## Examples (with code)

### Basic try/except
```python
try:
    num = int(input("Enter a number: "))
    result = 10 / num
    print(f"Result: {result}")
except ValueError:
    print("That's not a valid number!")
except ZeroDivisionError:
    print("Cannot divide by zero!")
```

### Multiple exceptions in one line
```python
try:
    data = [1, 2, 3]
    idx = int(input("Index: "))
    print(data[idx])
except (ValueError, IndexError) as e:
    print(f"Error: {e}")
```

### else and finally
```python
try:
    file = open("data.txt", "r")
    content = file.read()
except FileNotFoundError:
    print("File not found!")
else:
    print(f"Read {len(content)} characters")
finally:
    if 'file' in locals() and not file.closed:
        file.close()
        print("File closed.")
```

### raise to throw exceptions
```python
def withdraw(amount, balance):
    if amount <= 0:
        raise ValueError("Amount must be positive")
    if amount > balance:
        raise ValueError("Insufficient funds")
    return balance - amount

try:
    withdraw(500, 100)
except ValueError as e:
    print(f"Transaction failed: {e}")
```

### Custom Exceptions
```python
class InsufficientFundsError(Exception):
    """Raised when account balance is insufficient."""
    def __init__(self, balance, amount):
        self.balance = balance
        self.amount = amount
        super().__init__(f"Insufficient funds: need ${amount}, have ${balance}")

class NegativeAmountError(Exception):
    pass

class BankAccount:
    def __init__(self, owner, balance=0):
        self.owner = owner
        self.balance = balance

    def withdraw(self, amount):
        if amount <= 0:
            raise NegativeAmountError("Withdrawal amount must be positive")
        if amount > self.balance:
            raise InsufficientFundsError(self.balance, amount)
        self.balance -= amount
        return self.balance

try:
    acc = BankAccount("Alice", 100)
    acc.withdraw(200)
except InsufficientFundsError as e:
    print(f"Error: {e}")
except NegativeAmountError as e:
    print(f"Error: {e}")
else:
    print(f"New balance: {acc.balance}")
```

### Exception hierarchy
```python
# All built-in exceptions inherit from BaseException
# Common hierarchy:
# BaseException
#  +-- SystemExit
#  +-- KeyboardInterrupt

#       +-- ArithmeticError
#       |    +-- ZeroDivisionError
#       |    +-- OverflowError

# Catch general exceptions carefully — always catch specific when possible
try:
    x = int("hello")
except Exception:  # Too broad — hides unexpected errors
    print("Something went wrong")
```

### Re-raising exceptions
```python
def process_data(filename):
    try:
        with open(filename) as f:
            return f.read()
    except FileNotFoundError:
        print(f"File {filename} not found, creating it...")
        with open(filename, "w") as f:
            f.write("default data")
        return "default data"
    except PermissionError:
        print(f"No permission to read {filename}")
        raise  # Re-raise for the caller to handle
```

## Common mistakes

1. **Bare `except:` clause** — catches everything including `KeyboardInterrupt` and `SystemExit`, making it hard to stop the program.
   ```python
   # Bad
   try:
       x = 1 / 0
   except:  # Catches KeyboardInterrupt too!
       print("Error")
   
   # Good
   try:
       x = 1 / 0
   except Exception as e:
       print(f"Error: {e}")
   ```

2. **Catching too broad an exception** — hides bugs and makes debugging difficult.

3. **Empty except blocks** — silently swallows errors.
   ```python
   try:
       do_something()
   except:
       pass  # NEVER do this — you'll never know something failed
   ```

4. **Not using `finally` for cleanup** — leaving files or connections open on errors.

5. **Raising `Exception` directly** — creates ambiguity. Use specific or custom exceptions.

6. **Putting too much code in `try`** — makes it hard to know which line caused the error. Keep `try` blocks minimal.

7. **Forgetting that `else` runs only if no exception** — common confusion with `finally`.

8. **Catching `StopIteration` manually** — use for loops instead.

9. **Not handling `FileNotFoundError` separately from `PermissionError`** — they need different recovery logic.

10. **Creating custom exceptions without inheriting from `Exception`** — always inherit from `Exception`, not `BaseException`.

## Interview questions + answers

**Q1: What's the difference between `else` and `finally`?**
A: `else` runs only if no exception occurred in the `try` block. `finally` always runs regardless of exceptions — used for cleanup.

**Q2: Can you have multiple `except` blocks? How are they evaluated?**
A: Yes, they're evaluated top-to-bottom. The first matching `except` runs. Put specific exceptions before general ones, or the general one will catch everything first.

**Q3: What happens if an exception occurs in `except` or `finally`?**
A: In `except`, the new exception replaces the original unless you use `raise` with a cause (`raise NewException from original`). In `finally`, the new exception propagates; if one is already propagating, it may replace or be chained.

**Q4: Explain the exception hierarchy briefly.**
A: `BaseException` at the top. `SystemExit` and `KeyboardInterrupt` inherit directly from it. `Exception` (for all application errors) inherits from `BaseException`. All custom exceptions should inherit from `Exception`.

**Q5: What is `raise ... from ...` used for?**
A: It chains exceptions, preserving the original cause. Useful when converting one exception type to another.
```python
try:
    int("abc")
except ValueError as e:
    raise RuntimeError("Conversion failed") from e
```

**Q6: Can a `try` block have multiple `except` blocks for the same exception type?**
A: No. Use conditional logic inside one `except` block if you need different handling.

**Q7: When would you create a custom exception?**
A: When you need domain-specific error types (e.g., `InsufficientFundsError`, `ValidationError`) that carry extra data or need to be caught specifically by callers.

**Q8: What is an assertion error?**
A: `AssertionError` is raised by `assert` statements. Used for debugging/testing, not for runtime error handling. Can be disabled with `-O` flag.

## Knowledge check (5-10 questions with answers)

**Q1: What does this code print?**
```python
try:
    print("A")
    raise ValueError("test")
except ValueError:
    print("B")
else:
    print("C")
finally:
    print("D")
```
A: A, B, D (else doesn't run because exception occurred)

**Q2: What does this code print?**
```python
try:
    print("A")
except ValueError:
    print("B")
else:
    print("C")
finally:
    print("D")
```
A: A, C, D (no exception, so else runs)

**Q3: What does `except:` (bare except) catch?**
A: All exceptions including `SystemExit` and `KeyboardInterrupt`.

**Q4: Which is the base class for all built-in, non-system exceptions?**
A: `Exception`

**Q5: How do you access the exception message?**
A: Using `as e` then `str(e)` or `e.args`.

**Q6: What's wrong with `except ValueError, TypeError:`?**
A: That's Python 2 syntax. In Python 3, use parentheses: `except (ValueError, TypeError):`.

**Q7: Can `finally` contain a `return` statement? What happens?**
A: Yes. If `finally` returns, it overrides any exception or return from `try`/`except`. This is usually a bad idea.

**Q8: What's the difference between `raise` and `raise e`?**
A: `raise` (bare) re-raises the current exception with its original traceback. `raise e` raises a new reference to `e` and may reset the traceback. Bare `raise` preserves the original call stack.

**Q9: How do you create a custom exception with extra data?**
A: Create a class inheriting from `Exception` and add `__init__` with extra parameters.

**Q10: What is `try-except-else-finally` execution order when an exception is raised?**
A: try (until exception point) → except → finally (else is skipped)

## Revision notes (bullet points summary)

- `try` block contains risky code; `except` catches specific exceptions
- `else` runs only if no exception in `try`
- `finally` always runs — use for cleanup (close files, release resources)
- `raise` explicitly throws an exception; `raise` (bare) re-raises current exception
- Custom exceptions inherit from `Exception`; name them with `Error` suffix
- Always catch specific exceptions, never bare `except:`
- Use `as e` to access the exception object
- Exception hierarchy: `BaseException` → `Exception` → specific errors
- `with` statement (context manager) handles cleanup automatically for files, locks
- Common exceptions: `ValueError`, `TypeError`, `IndexError`, `KeyError`, `FileNotFoundError`, `ZeroDivisionError`
- Keep `try` blocks minimal — only wrap the code that might fail
- `raise ... from ...` chains exceptions to preserve cause
- Never use empty `except: pass` — you'll silently swallow errors

### Creative Mini Project: Safe Division Calculator with Error Handling

```python
"""
Safe Division Calculator
A calculator that handles all edge cases gracefully.
"""

class DivisionError(Exception):
    """Custom exception for division-specific errors."""
    pass

class SafeCalculator:
    def __init__(self, name="SafeCalc"):
        self.name = name
        self.history = []
        self.log_file = "calc_log.txt"

    def log(self, message):
        with open(self.log_file, "a") as f:
            f.write(f"{message}\n")

    def check_number(self, value):
        """Validate input is a number."""
        if not isinstance(value, (int, float)):
            raise TypeError(f"Expected a number, got {type(value).__name__}")
        if isinstance(value, float) and (value != value):  # NaN check
            raise ValueError("Value is NaN (not a number)")

    def divide(self, a, b):
        """Safe division with comprehensive error handling."""
        try:
            self.check_number(a)
            self.check_number(b)
            
            # Check for overflow
            if abs(a) > 1e308 or abs(b) > 1e308:
                raise OverflowError("Number too large")
            
            if b == 0:
                raise DivisionError("Cannot divide by zero")
            
            if isinstance(a, complex) or isinstance(b, complex):
                raise TypeError("Complex numbers not supported")
            
            result = a / b
            
            # Check result for infinity (float overflow)
            if result == float('inf') or result == float('-inf'):
                raise OverflowError("Result too large")
            
            entry = f"{a} / {b} = {result}"
            self.history.append(entry)
            self.log(f"SUCCESS: {entry}")
            return result
            
        except TypeError as e:
            error_msg = f"TYPE ERROR: {e}"
            self.log(error_msg)
            print(f"❌ {error_msg}")
            print("   Hint: Use only integers or floats.")
            
        except ZeroDivisionError:
            # Built-in exception (though caught by DivisionError first)
            error_msg = f"DIVISION ERROR: {a} / {b} — zero divisor"
            self.log(error_msg)
            print(f"❌ Cannot divide {a} by zero.")
            
        except DivisionError as e:
            error_msg = f"DIVISION ERROR: {e}"
            self.log(error_msg)
            print(f"❌ {e}")
            
        except OverflowError as e:
            error_msg = f"OVERFLOW: {e}"
            self.log(error_msg)
            print(f"❌ {e}")
            
        except Exception as e:
            error_msg = f"UNEXPECTED ERROR: {type(e).__name__}: {e}"
            self.log(error_msg)
            print(f"❌ An unexpected error occurred: {e}")
            
        return None

    def divide_chain(self, value, *divisors):
        """Chain multiple divisions: value / d1 / d2 / ..."""
        result = value
        for d in divisors:
            result = self.divide(result, d)
            if result is None:
                print("  Chain stopped due to error.")
                return None
        return result

    def show_history(self):
        """Display calculation history."""
        print(f"\n=== {self.name} History ===")
        if not self.history:
            print("No calculations yet.")
        for entry in self.history:
            print(f"  • {entry}")
        print("=" * 30)

    def show_log(self):
        """Show log file contents."""
        try:
            with open(self.log_file) as f:
                content = f.read()
            if content.strip():
                print("=== Log Contents ===")
                print(content)
            else:
                print("Log is empty.")
        except FileNotFoundError:
            print("No log file found.")
        except PermissionError:
            print("Cannot read log file (permission denied).")


def interactive_mode():
    """Interactive calculator mode with input validation."""
    calc = SafeCalculator()
    
    print("=" * 40)
    print("  SAFE DIVISION CALCULATOR")
    print("  Type 'quit' to exit, 'history' for history")
    print("=" * 40)
    
    while True:
        try:
            user_input = input("\nEnter: a / b : ").strip()
            
            if user_input.lower() == 'quit':
                print("Goodbye!")
                break
            if user_input.lower() == 'history':
                calc.show_history()
                continue
            if user_input.lower() == 'log':
                calc.show_log()
                continue
            if not user_input:
                continue
            
            # Parse input
            parts = user_input.split('/')
            if len(parts) != 2:
                print("  Format: number / number")
                continue
            
            a = float(parts[0].strip())
            b = float(parts[1].strip())
            
            result = calc.divide(a, b)
            if result is not None:
                print(f"✅ Result: {result}")
                
        except ValueError:
            print("❌ Invalid number format. Enter numeric values.")
        except KeyboardInterrupt:
            print("\nGoodbye!")
            break
        except Exception as e:
            print(f"❌ Unexpected input error: {e}")


# --- Test cases ---
if __name__ == "__main__":
    # Run tests
    calc = SafeCalculator()
    
    print("=== Test Suite ===\n")
    
    # Normal division
    print("1. Normal:", calc.divide(10, 3))          # 3.333...
    
    # Integer division
    print("2. Integers:", calc.divide(15, 4))        # 3.75
    
    # Division by zero
    print("3. Zero division:", calc.divide(10, 0))   # Error
    
    # String input (TypeError)
    print("4. String:", calc.divide("10", 2))        # Error
    
    # Negative numbers
    print("5. Negative:", calc.divide(-10, 3))       # -3.333...
    
    # Float precision
    print("6. Float:", calc.divide(0.1, 0.2))        # 0.5
    
    # Division chain
    print("\n=== Chain Division ===")
    calc.divide_chain(100, 2, 5, 0, 10)  # Stops at zero
    
    # History
    calc.show_history()
```
