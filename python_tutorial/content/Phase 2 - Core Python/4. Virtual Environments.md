# 4. Virtual Environments

## Definition
A virtual environment is an isolated Python environment that has its own installation directories, Python interpreter, and set of installed packages. It doesn't share packages with the system Python or other virtual environments. Created with the built-in `venv` module: `python -m venv myenv`.

## Why it exists
Different projects need different dependencies, and those dependencies often conflict:
- **Project A** needs `Django==4.2` but **Project B** needs `Django==5.0`
- **System packages** should never be modified — breaking system Python breaks system tools
- **Version conflicts**: Installing one package upgrades a shared dependency and breaks another project
- **Reproducibility**: Virtual environments combined with `requirements.txt` let you exactly reproduce any project's setup
- **Clean uninstall**: Delete the virtual environment directory and it's gone — no leftover packages

Without virtual environments, you'd need to install everything globally, leading to "dependency hell."

## How it works

### Creating a virtual environment
```bash
# Create a virtual environment named 'venv' or '.venv'
python -m venv venv
python -m venv .venv
```

### Activating and deactivating
```bash
# macOS / Linux
source venv/bin/activate
source .venv/bin/activate

# Windows (Command Prompt)
venv\Scripts\activate.bat
.venv\Scripts\activate

# Windows (PowerShell)
venv\Scripts\Activate.ps1
.venv\Scripts\Activate.ps1

# Deactivate (all platforms)
deactivate
```

When activated, your shell prompt changes to show the environment name:
```bash
(venv) user@machine:~/myproject$
```

### Working inside a virtual environment
```bash
# After activation, pip installs inside the environment
pip install requests
pip install numpy pandas matplotlib

# Freeze to save dependencies
pip freeze > requirements.txt

# Exit the environment
deactivate

# Later, another developer recreates it:
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt
```

## Internal working

### What happens when you create a virtual environment
`python -m venv myenv` creates this structure:

```
myenv/
├── bin/                     # (or Scripts\ on Windows)
│   ├── python              # Symlink/copy of Python interpreter
│   ├── pip                 # pip for this environment
│   ├── activate            # Activation script (bash)
│   ├── activate.csh        # Activation script (csh)
│   ├── activate.fish       # Activation script (fish)
│   └── deactivate          # Deactivation script (bash)
├── include/                 # Python header files
├── lib/
│   └── python3.x/
│       └── site-packages/  # Third-party packages go here
├── pyvenv.cfg              # Configuration file
└── share/                  # Man pages, etc.
```

### How activation works
When you run `source venv/bin/activate`:
1. It modifies `PATH` to prepend `venv/bin/`, so `python` and `pip` point to the virtual environment's versions
2. It sets `VIRTUAL_ENV` environment variable pointing to the environment directory
3. It changes the shell prompt to show `(venv)`
4. `deactivate` is defined as a shell function that reverses these changes

### How isolation works
- The virtual environment's `python` binary is a symlink to the system Python (or a copy)
- `site-packages/` directory inside `lib/` stores all `pip install`-ed packages
- `sys.path` is modified so the environment's `site-packages` is searched before the system's
- The `pyvenv.cfg` file has `include-system-site-packages = false` (by default) to prevent system package leakage

## Real-world analogy
A **hotel room** vs. **your home**:
- Your home (system Python) has all your stuff (packages) — but you can't reorganize it for every guest
- Each virtual environment is like a **hotel room** — you check in, personalize it with exactly what you need for your stay (project), use it, and check out (delete the directory)
- No two hotel rooms interfere with each other
- When you check out, everything is cleanly removed — no mess left behind
- `requirements.txt` is like a packing list that lets you set up identical rooms for future stays

```bash

```

### pip freeze vs pip list
```bash
# pip list — human-readable format
pip list
# Package    Version
# ---------- -------
# numpy      1.24.0
# requests   2.31.0

# pip freeze — machine-parsable (requirements.txt format)
pip freeze
# numpy==1.24.0
# requests==2.31.0
```

### Managing multiple environments
```bash
# Check which environment is active (if any)
echo $VIRTUAL_ENV
# /home/user/myblog/.venv

# Or in Python code:
import sys
print(sys.prefix)  # Path to current environment
print(sys.base_prefix)  # Path to system Python
```

## Common mistakes

1. **Forgetting to activate the environment** — pip installs globally instead. Always check `(venv)` in your prompt.

2. **Committing the virtual environment** — Never commit `venv/` or `.venv/` to version control. Add to `.gitignore`.

3. **Using `sudo pip install`** — Never use `sudo` with pip. It installs globally and can break system packages. Always use a virtual environment.

4. **Not running `pip freeze` immediately** — You'll forget what you installed. Run `pip freeze > requirements.txt` right after installing.

5. **`pip freeze` dumping everything** — If you activated the environment but forgot to install requirements first, you might miss listings. Always freeze in the active environment.

6. **Confusing `python -m venv venv`** — The first `venv` is the module, the second is the directory name. They're different things.

7. **Not using `.venv` naming convention** — Many tools (VS Code, pre-commit) look for `.venv` by default. Using `.venv` as the directory name is a best practice.

8. **Installing packages before activating** — If you install while not activated, packages go to the system. Then when you activate, they aren't available.

9. **Deleting the environment directory** — If you delete `.venv/` without `pip freeze`, you lose your dependency list. Always freeze first.

10. **Using environment-specific paths in code** — Don't hardcode paths like `.venv/lib/python3.x/site-packages`. Everything should use relative imports.

## Interview questions + answers

**Q1: What is a virtual environment and why use one?**
A: An isolated Python environment with its own packages, separate from the system Python. It prevents dependency conflicts between projects and ensures reproducibility.

**Q2: How do you create and activate a virtual environment?**
A: `python -m venv .venv` then `source .venv/bin/activate` (Linux/macOS) or `.venv\Scripts\activate` (Windows).

**Q3: What's the difference between `pip freeze` and `pip list`?**
A: `pip list` shows packages in a table format. `pip freeze` outputs packages in `name==version` format suitable for `requirements.txt`.

**Q4: Should you commit the virtual environment to version control?**
A: No. Commit `requirements.txt` instead. Virtual environments are platform-specific and large. Add `.venv/` to `.gitignore`.

**Q5: How do you recreate a virtual environment from requirements.txt?**
A: `python -m venv .venv && source .venv/bin/activate && pip install -r requirements.txt`

**Q6: What happens when you activate a virtual environment?**
A: It modifies `PATH` to point to the environment's `bin/` directory, sets `VIRTUAL_ENV` variable, and changes the shell prompt.

**Q7: Can you have multiple virtual environments for the same project?**
A: Yes, e.g., one for Python 3.10 and one for Python 3.11 to test compatibility.

**Q8: How do you delete a virtual environment?**
A: Simply delete the directory: `rm -rf .venv`. No uninstall commands needed.

**Q9: What's the recommended name for a virtual environment directory?**
A: `.venv` (dot-prefixed so it's hidden, and commonly recognized by tools).

**Q10: What is `pyvenv.cfg`?**
A: The configuration file inside a virtual environment. It contains paths and settings like `include-system-site-packages`.

## Revision notes (bullet points summary)

- Virtual environments isolate project dependencies from each other and from system Python
- Create: `python -m venv .venv`
- Activate: `source .venv/bin/activate` (Linux/macOS) or `.venv\Scripts\activate` (Windows)
- Deactivate: `deactivate`
- `pip freeze > requirements.txt` saves exact dependency versions
- `pip install -r requirements.txt` reproduces the environment
- Never commit `.venv/` — add to `.gitignore`
- Never use `sudo pip install` — always use a virtual environment
- `VIRTUAL_ENV` environment variable tracks the active environment
- Use `.venv` as the standard naming convention
- Each environment has its own `bin/`, `lib/`, and `site-packages/`
- Delete by removing the directory — no cleanup needed
- `pip list` shows packages; `pip freeze` outputs requirements format

cat > src/main.py << 'PYTHON'
"""
Data Analyzer — Main entry point.
"""

import sys
import click
import pandas as pd
import matplotlib.pyplot as plt
from pathlib import Path

@click.command()
@click.option('--input', '-i', required=True, help='Input CSV file')
@click.option('--output', '-o', default='reports/', help='Output directory')
@click.option('--show', is_flag=True, help='Show plot window')
def main(input, output, show):
    """Analyze a CSV dataset."""
    input_path = Path(input)
    output_path = Path(output)
    output_path.mkdir(parents=True, exist_ok=True)

    if not input_path.exists():
        click.echo(f"Error: File {input_path} not found", err=True)
        sys.exit(1)

    click.echo(f"Loading {input_path}...")
    df = pd.read_csv(input_path)

    click.echo(f"\nDataset shape: {df.shape}")
    click.echo(f"Columns: {list(df.columns)}")

    # Summary statistics
    summary = df.describe()
    summary_path = output_path / "summary.csv"
    summary.to_csv(summary_path)
    click.echo(f"Summary saved to {summary_path}")

    # Generate plots for numeric columns
    numeric_cols = df.select_dtypes(include='number').columns
    for col in numeric_cols[:5]:  # Limit to first 5
        fig, ax = plt.subplots()
        df[col].hist(ax=ax, bins=30)
        ax.set_title(f"Distribution of {col}")
        plot_path = output_path / f"{col}_histogram.png"
        plt.savefig(plot_path, dpi=100, bbox_inches='tight')
        click.echo(f"Plot saved to {plot_path}")
        if show:
            plt.show()
        plt.close()

    click.echo("\nAnalysis complete!")

if __name__ == "__main__":
    main()
PYTHON
