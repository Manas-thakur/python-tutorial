# 3. Packages

## Definition
A package is a way to organize related modules into a directory hierarchy. A package is simply a directory containing a special `__init__.py` file (which can be empty) and one or more module files (`.py`). Packages allow for namespacing and structuring larger applications. Third-party packages are distributed via **pip** (Python's package installer) from **PyPI** (Python Package Index).

## Why it exists
- **Organization**: Group related modules under a common namespace (e.g., `email.mime.text`, `os.path`)
- **Scalability**: Flat module listings become messy in large projects. Packages provide hierarchy
- **Namespacing**: `package.module.function()` is clearer and avoids naming conflicts
- **Distribution**: Packages are the standard unit for distributing reusable code via PyPI
- **Third-party ecosystem**: pip and PyPI enable installing millions of packages with one command

## How it works

### Package structure
```
mypackage/
├── __init__.py          # Makes 'mypackage' a package; runs on import
├── module_a.py          # Regular module
├── module_b.py          # Regular module
├── subpackage/          # Sub-package (nested directory)
│   ├── __init__.py
│   └── module_c.py
└── utils/
    ├── __init__.py
    └── helpers.py
```

### Importing from packages
```python
# Absolute imports
import mypackage.module_a
from mypackage import module_b
from mypackage.module_a import some_function
from mypackage.subpackage.module_c import ClassName
import mypackage.utils.helpers as helpers

# Relative imports (inside a package)
# In mypackage/module_a.py:
from . import module_b          # Import sibling module
from .subpackage import module_c  # Import from subpackage
from ..utils import helpers      # Import from parent package
```

## Internal working

### The `__init__.py` file
When a package is imported (`import mypackage`), Python executes `__init__.py`. This file:
- **Initializes** the package  -  runs setup code, imports key classes/functions
- **Defines `__all__`**  -  controls what `from package import *` exports
- **Imports submodules**  -  makes them accessible as `package.module`
- It can be empty; Python 3.3+ supports **namespace packages** (packages without `__init__.py`)

```python
# Inside math_package/__init__.py
from .basic import add, subtract
from .advanced import power, sqrt
from .geometry import circle_area

__all__ = ['add', 'subtract', 'power', 'sqrt', 'circle_area']
```

### Absolute vs relative imports
- **Absolute**: `from package.subpackage import module`  -  full path from top-level
- **Relative**: `from . import sibling`  -  relative to current module's position
  - `.` = current package
  - `..` = parent package
  - `...` = grandparent package
- Relative imports only work inside packages (not in top-level scripts)

### sys.path and package discovery
When you `import mypackage`, Python searches `sys.path` for a directory named `mypackage` containing `__init__.py`. If found, it loads the package. This is the same mechanism as module loading.

## Real-world analogy
An **office filing cabinet**:
- The cabinet itself is the **package** (`accounts/`, `hr/`, `engineering/`)
- Each drawer is a **subpackage** (`accounts/receivable/`, `accounts/payable/`)
- Each folder inside a drawer is a **module** (`accounts/receivable/invoices.py`)
- The `__init__.py` is like the drawer's label  -  it tells you this drawer belongs to this cabinet
- **pip** is like a courier service that delivers new filing cabinets from a warehouse (PyPI) to your office
- `requirements.txt` is like a shopping list of cabinets you need

## Common mistakes

1. **Forgetting `__init__.py`**  -  in Python 3.3+, packages without `__init__.py` become namespace packages, which can cause subtle issues. Always include it.

2. **Circular imports within a package**  -  package A imports from B, B imports from A. Causes `ImportError`.

3. **Using relative imports in scripts**  -  relative imports only work inside packages, not from top-level scripts. Run packages with `python -m mypackage.main`.

4. **`from package import *` without `__all__`**  -  if `__all__` isn't defined in `__init__.py`, only names not starting with `_` are imported. Define `__all__` explicitly.

5. **Installing packages globally without virtual environments**  -  causes conflicts between projects. Always use `venv`.

6. **Not pinning versions in requirements.txt**  -  `requests` instead of `requests==2.31.0`. Future updates may break code.

7. **Committing `__pycache__` and `.pyc` files**  -  add `__pycache__/` and `*.pyc` to `.gitignore`.

8. **Deep nesting**  -  packages nested more than 3-4 levels deep become hard to use and import.

9. **Mixed absolute and relative imports**  -  pick one style (absolute is more readable) and be consistent.

10. **Not running `pip freeze` after setup**  -  you'll forget which packages you installed. Run `pip freeze > requirements.txt` immediately.

## Interview questions + answers

**Q1: What's the difference between a module and a package?**
A: A module is a single `.py` file. A package is a directory containing `__init__.py` and one or more modules (or sub-packages). Packages can contain packages.

**Q2: What is `__init__.py` and its purpose?**
A: It's executed when a package is imported. It initializes the package, imports submodules, defines `__all__`, and can run setup code. In Python 3.3+, it's optional for namespace packages.

**Q3: What's the difference between absolute and relative imports?**
A: Absolute imports specify the full path from the top-level package (`from pkg.sub.module import X`). Relative imports use dots to reference the current (`.`) or parent (`..`) package relative to the current module.

**Q4: How do you install a package from requirements.txt?**
A: `pip install -r requirements.txt`

**Q5: What's the purpose of `__all__`?**
A: It controls which names are exported when `from package import *` is used. Without it, `*` imports all public names (no underscore prefix).

**Q6: What is PyPI?**
A: The Python Package Index  -  the official repository of third-party Python packages. pip downloads from PyPI by default.

**Q7: What does `pip freeze` do?**
A: Lists all installed packages and their versions in `requirements.txt` format. Used to reproduce environments.

**Q8: Can you run a package as a script?**
A: Yes. Create a `__main__.py` file in the package, then run with `python -m mypackage`.

## Knowledge check (5-10 questions with answers)

**Q1: What file makes a directory a Python package?**
A: `__init__.py` (or being a namespace package in Python 3.3+)

**Q2: What does `from . import something` mean?**
A: Import `something` from the current package (relative import).

**Q3: What command installs all packages from requirements.txt?**
A: `pip install -r requirements.txt`

**Q4: What does `pip freeze` output?**
A: A list of installed packages with exact version numbers, in `name==version` format.

**Q5: Can you have nested packages?**
A: Yes. A package directory can contain subdirectories with their own `__init__.py`.

**Q6: What happens if two packages have the same name?**
A: Python loads the first one found in `sys.path`. This is why virtual environments are important.

**Q7: How do you create a package that runs as `python -m mypackage`?**
A: Add a `__main__.py` file in the package that handles the entry point.

**Q8: What command shows all installed pip packages?**
A: `pip list`

**Q9: Is `__init__.py` required in Python 3.3+?**
A: No, but it's recommended. Without it, the directory is a namespace package.

**Q10: Which is better  -  absolute or relative imports?**
A: Absolute imports are generally preferred for readability and clarity. Relative imports are useful within deeply nested packages to avoid verbose paths.

## Revision notes (bullet points summary)

- A **package** is a directory with `__init__.py` containing modules/sub-packages
- `__init__.py` runs on import  -  use it for initialization and importing key names
- `__all__` controls what `from package import *` exports
- **Absolute imports**: `from package.sub.module import name` (preferred)
- **Relative imports**: `from . import sibling`, `from .. import parent` (use sparingly)
- **pip** installs packages from **PyPI**  -  `pip install package_name`
- `pip list` shows installed; `pip freeze` generates requirements.txt format
- `requirements.txt` lists dependencies with version pins (`numpy==1.24.0`)
- `pip install -r requirements.txt` reproduces an environment
- Always use virtual environments to isolate project dependencies
- Popular packages: `requests`, `numpy`, `pandas`, `matplotlib`, `flask`, `django`
- Add `__pycache__/` and `*.pyc` to `.gitignore`

```python
def log(x, base=10):
```

```python
# ============================================================
# calculator_pkg/utils/helpers.py
# ============================================================
def round_to(value, decimals=2):
    return round(value, decimals)

def percentage(part, total):
    if total == 0:
        raise ValueError("Total cannot be zero")
    return (part / total) * 100

def clamp(value, min_val, max_val):
    return max(min_val, min(value, max_val))
```
